# Plot-Werkstatt (Beat-Board)

Planendes Pendant zur rückwärtsgewandten Szenen-/Ereignis-Analyse: Der User skizziert die Handlung seines Buches als Kanban-artiges Beat-Board — **Akte** als Spalten (`plot_acts`), **Beats** (einzelne Handlungspunkte) als Karten darin (`plot_beats`). KI assistiert ausschliesslich **planend/überwachend** (Brainstorm + Consistency gegen die Buchrealität), schreibt nie Fliesstext ins Manuskript (App-Philosophie: KI nur rückwärtsgewandt/weltaufbauend, nie generativ in den Buchtext).

**Optionale zweite Ordnungsachse — Handlungsstränge (Swimlanes, `plot_threads`):** Für grosse Multi-POV-Bücher wird das Board zum Raster **Akte (Spalten) × Stränge (Zeilen)**; ein Beat sitzt in der Zelle `(act_id, thread_id)`. Ein Strang ist optional an eine Hauptfigur gebunden (Katalog **oder** Werkstatt). **Opt-in / null Stränge = flaches Board:** ohne Strang rendert die flache Akt-Spaltenansicht unverändert ([plot-board-flat.html](../public/partials/plot-board-flat.html)); der erste Strang schaltet aufs Grid um ([plot-board-grid.html](../public/partials/plot-board-grid.html)). Die „ohne Strang"-Lane (`thread_id = NULL`) fängt alle nicht zugeordneten Beats.

Pro **Buch + User** skopiert (lokaler Karten-State, nicht im `catalog-store` — Plot ist nicht mit Figuren/Orten geteilt). Code: [db/plot.js](../db/plot.js), [routes/plot.js](../routes/plot.js), [routes/jobs/plot.js](../routes/jobs/plot.js), [public/js/cards/plot-card.js](../public/js/cards/plot-card.js), [public/js/book/plot.js](../public/js/book/plot.js), [public/partials/plot.html](../public/partials/plot.html) (Rahmen) + [plot-board-flat.html](../public/partials/plot-board-flat.html) / [plot-board-grid.html](../public/partials/plot-board-grid.html) / [plot-thread-bar.html](../public/partials/plot-thread-bar.html) (nested Partials) + [plot-beat-cell.html](../public/partials/plot-beat-cell.html) (geteiltes Beat-Markup via Fragment-Include), [public/css/book/plot/board.css](../public/css/book/plot/board.css) + [plot/widgets.css](../public/css/book/plot/widgets.css) + [plot/swimlane.css](../public/css/book/plot/swimlane.css), [public/js/prompts/plot.js](../public/js/prompts/plot.js).

Trigger: `tile.plot` (Quick-Pill / Palette-Aliase `handlung|beat|board|akt|struktur|outline|dramaturgie|story|wendepunkt`). Hash-Permalink `#book/:bookId/plot[/:beatId]` — der optionale `beatId` ist der gerade bearbeitete Beat (Root-SSoT `plotBeatId`, spiegelt `editingBeatId`); Deep-Link öffnet die Karte + fokussiert den Beat (Edit + zentriert ins Bild via `plot:focus-beat`-Event / `_pendingFocusBeatId` bis das Board geladen ist). Gruppe `world`, `minRole: 'editor'` (planendes Werkzeug, kein Lesezugang).

## Datenmodell

```
plot_acts (id, book_id→books CASCADE, user_email, name, farbe,
           thread_id→plot_threads CASCADE, position, created_at, updated_at)
plot_threads (id, book_id→books CASCADE, user_email, name, farbe,
              figure_id→figures SET NULL, draft_figure_id→draft_figures SET NULL,
              chapter_id→chapters SET NULL,
              position, created_at, updated_at)
   └── 1:N plot_beats (id, book_id→books CASCADE, act_id→plot_acts CASCADE,
                       thread_id→plot_threads SET NULL, user_email,
                       titel, beschreibung, status, chapter_id→chapters SET NULL,
                       intensitaet, sort_order, created_at, updated_at)
          ├── M:N plot_beat_figures (beat_id→plot_beats CASCADE, figure_id→figures CASCADE,
          │                          PK(beat_id, figure_id))
          └── M:N plot_beat_draft_figures (beat_id→plot_beats CASCADE,
                                           draft_figure_id→draft_figures CASCADE,
                                           PK(beat_id, draft_figure_id))
```

- **`plot_acts`** — Board-Spalten (Akte/Phasen), geordnet via `position` **pro Scope** (geteilt vs. je Strang eine eigene 0..n-Sequenz). `farbe` optional (max 32 chars). **Hybrid-Akte (`thread_id`, FK → `plot_threads` CASCADE):** `NULL` = **geteilter** Akt (flaches Board + alle Stränge ohne eigene Aktstruktur); gesetzt = der Akt gehört **nur** Strang `T`. „Strang hat eigene Akte" wird allein aus der Existenz strang-eigener Akte abgeleitet (kein Flag, kein Drift). `forkThreadActs` klont die geteilten Akte in den Strang + hängt dessen Beats auf die Klone um; `unforkThreadActs` hängt sie positionsweise zurück (Überzahl → letzte geteilte Spalte) + löscht die eigenen Akte. Invariante: ein Beat sitzt nur auf einem Akt, der **geteilt** ist oder **seinem eigenen Strang** gehört (`ACT_THREAD_MISMATCH` sonst).
- **`plot_threads`** — Board-Zeilen (Handlungsstränge / Swimlanes), geordnet via `position`. `farbe` = Palette-Key (analog `plot_acts.farbe`). Optionale Hauptfigur-Bindung: `figure_id` (FK → `figures.id`, INTEGER; nach aussen als TEXT-`fig_id` exponiert via `_THREAD_SELECT`-JOIN) **oder** `draft_figure_id` (FK → `draft_figures.id`, INTEGER, bereits Frontend-Identität). Beide SET NULL — Figur löschen entkoppelt nur die Bindung. Ein Strang darf auch ungebunden sein (z. B. „B-Story"). Optionale Kapitel-Bindung: `chapter_id` (FK → `chapters` SET NULL; `chapter_name` via `_THREAD_SELECT`-JOIN). **Live-Vererbung Strang → Beat:** ein Beat einer Strang-Lane erbt implizit die Hauptfigur des Strangs (immer als beteiligt) und — sofern er kein eigenes `chapter_id` hat — dessen Kapitel. Die geerbten Werte werden **nie** auf den Beat geschrieben (nur Anzeige im Grid als gedämpfte/gepunktete Badges via `inheritedFigureForBeat`/`inheritedChapterForBeat` + KI-Kontext); Strang-Bindung ändern → alle Beats der Lane folgen automatisch.
- **`plot_beats`** — Karten in einer Zelle (Akt × Strang), geordnet via `sort_order` **pro Zelle** (`act_id` + `thread_id`). `thread_id` (SET NULL) ist die Strang-Zuordnung (`NULL` = „ohne Strang"-Lane; null Stränge → flaches Board). `status ∈ {geplant, im_buch}` (CHECK-Constraint) ist die binäre Realisierungsachse (Idee ↔ eingearbeitet). `verworfen` (0/1, CHECK) ist eine eigene, orthogonale Verwerfen-Achse — bleibt bei Status-Wechsel erhalten, hält verworfene Beats aus Spannungsbogen/Coverage/aktiver Verteilung raus. `chapter_id` (SET NULL) verknüpft den Beat mit dem Zielkapitel im Manuskript.
- **`plot_beat_figures`** — M:N-Brücke Beat ↔ Katalog-Figur (`figures.id`, INTEGER-FK), welche Figuren ein Handlungspunkt involviert. Nach aussen wird die TEXT-`fig_id` exponiert (Frontend-Identität), Schreib-/Lesepfad übersetzt via `resolveFigureIds`.
- **`plot_beat_draft_figures`** — parallele M:N-Brücke Beat ↔ Werkstatt-Figur (`draft_figures.id`, INTEGER-FK). Anders als bei Katalog-Figuren IST die `draft_figures.id` bereits die Frontend-Identität (keine TEXT-Indirektion). So kann ein Beat sowohl bereits extrahierte Buch-Figuren als auch vorwärts-entwickelte, evtl. noch nicht im Manuskript stehende Werkstatt-Figuren involvieren.
- **`plot_beats.intensitaet`** — optionaler Spannungswert 1–5 (CHECK `NULL OR BETWEEN 1 AND 5`), speist den Spannungsbogen. `NULL` = nicht gesetzt; verworfene Beats fliessen nicht in den Bogen.
- **`plot_consistency_runs`** — Historie der Konsistenz-Prüfungen, pro (Buch, User) skopiert: `book_id` (FK → `books` CASCADE), `created_at`, `konflikt_count` (denormalisiert fürs Listen-Rendering), `result_json` (`{ konflikte, fazit }`), `model`. `ON DELETE CASCADE` — die Historie stirbt mit dem Buch. Anders als die Figuren-Werkstatt (`werkstatt_runs` pro Draft) bezieht sich Plot auf das Buch, nicht auf einen Draft.
- **`plot_brainstorm_runs`** — Historie der Brainstorm-Läufe, pro (Buch, User) skopiert **und** zusätzlich pro `act_id` (FK → `plot_acts` SET NULL) / `thread_id` (FK → `plot_threads` SET NULL): `book_id` (CASCADE), `created_at`, `vorschlag_count` (denormalisiert), `result_json` (`{ vorschlaege }`), `model`. **act_id/thread_id SET NULL** — ein gelöschter Akt/Strang entkoppelt den Lauf nur; `act_name`/`thread_name` kommen zur Lesezeit per JOIN (kein Snapshot, fallen bei gelöschtem Akt auf `null`). Nur Läufe mit ≥1 Vorschlag werden persistiert.
- Migration **184** (Akte/Beats/Katalog-Brücke) + **185** (`plot_beat_draft_figures`) + **186** (`plot_beats.intensitaet`, additives ADD COLUMN) + **187** (`plot_threads` + `plot_beats.thread_id`, additiv) + **191** (`plot_consistency_runs`, additiv) + **192** (`plot_threads.chapter_id`, additiv) + **193** (`plot_acts.thread_id` — Hybrid-Akte, additiv) + **204** (`plot_brainstorm_runs`, additiv) ([db/migrations.js](../db/migrations.js)). ## Routen (CRUD)

Alle unter `/plot` ([routes/plot.js](../routes/plot.js)), ACL via `requireBookAccess(req, bookId, 'editor')`. Akt-/Beat-Mutationen prüfen zusätzlich `user_email`-Owner.

| Methode | Pfad | Zweck |
|---------|------|-------|
| `GET`    | `/?book_id=X`      | Board laden → `{ acts, threads, beats }`. Beats inkl. `chapter_name` (LEFT JOIN) + `thread_id` + `fig_ids[]`. Threads inkl. `fig_id` (TEXT, via JOIN) + `draft_figure_id` + `chapter_id` + `chapter_name` (LEFT JOIN) |
| `GET`    | `/figure-usage?book_id=X&draft_id=Y` | Plot-Beteiligung einer Werkstatt-Figur → `{ beatCount, activeBeatCount, threads }` (via `figurePlotUsage`; `source_figure_id` serverseitig aus dem Owner-geprüften Draft). Speist das Cross-Feature-Badge in der [Figuren-Werkstatt](figur-werkstatt.md) |
| `GET`    | `/page-beat-counts?book_id=X`    | Map `page_id` → Anzahl nicht-verworfener Beats im Kapitel der Seite (`pageBeatCounts`). Speist den Plot-Verknüpfungs-Indikator im Notebook-Editor (analog `/ideen/counts`, `/research/page-counts`) |
| `GET`    | `/chapter-beat-counts?book_id=X` | Map `chapter_id` → Anzahl nicht-verworfener Beats im Kapitel (`chapterBeatCounts`). Speist den Indikator in der Kapitelansicht |
| `POST`   | `/acts`            | `{ book_id, name, farbe?, thread_id? }` — neuer Akt ans Spaltenende (gesetzter `thread_id` → strang-eigener Akt, Hybrid; sonst geteilt) |
| `PATCH`  | `/acts/:id`        | `{ name?, farbe? }` umbenennen/umfärben |
| `DELETE` | `/acts/:id`        | Akt löschen — `plot_beats` hängen via CASCADE dran |
| `PUT`    | `/acts/order`      | `{ book_id, order:[actId,…] }` — Spalten-Reihenfolge (`position`) |
| `POST`   | `/threads`         | `{ book_id, name, farbe?, figure_id?, draft_figure_id?, chapter_id? }` — neuer Strang ans Zeilenende |
| `PATCH`  | `/threads/:id`     | `{ name?, farbe?, figure_id?, draft_figure_id?, chapter_id? }` — Figuren-/Kapitel-Bindung nur bei explizitem Key geändert |
| `DELETE` | `/threads/:id`     | Strang löschen — `plot_beats.thread_id` fällt via SET NULL auf „ohne Strang" |
| `PUT`    | `/threads/order`   | `{ book_id, order:[threadId,…] }` — Zeilen-Reihenfolge (`position`) |
| `POST`   | `/threads/:id/fork-acts`   | Eigene Aktstruktur aktivieren: geteilte Akte in den Strang klonen + dessen Beats umhängen (idempotent; `NO_SHARED_ACTS` ohne geteilte Akte) |
| `DELETE` | `/threads/:id/fork-acts`   | Eigene Aktstruktur auflösen: Beats positionsweise zurück auf die geteilten Akte, eigene Akte löschen |
| `POST`   | `/beats`           | `{ book_id, act_id, titel, beschreibung?, status?, chapter_id?, thread_id?, figure_ids?, draft_figure_ids? }` |
| `PATCH`  | `/beats/:id`       | Partielles Update (nur übergebene Felder, inkl. `thread_id`); `figure_ids`/`draft_figure_ids` (Array) ersetzen die jeweiligen Links komplett |
| `DELETE` | `/beats/:id`       | Beat löschen |
| `PUT`    | `/beats/order`     | `{ book_id, order:[{actId, threadId, beatIds:[…]}] }` — setzt `act_id` + `thread_id` + `sort_order` für DnD (fehlendes `threadId` → NULL) |
| `GET`    | `/consistency-runs?book_id=X` | Konsistenz-Prüfungs-Historie → Liste (ohne `result_json`, mit `konflikt_count`), `created_at DESC` |
| `GET`    | `/consistency-runs/:id`       | Lauf-Detail inkl. geparstem `result` (`{ konflikte, fazit }`) |
| `DELETE` | `/consistency-runs/:id`       | Einzel-Lauf löschen (Owner via `user_email`) |
| `GET`    | `/brainstorm-runs?book_id=X`  | Brainstorm-Lauf-Historie → Liste (ohne `result_json`, mit `vorschlag_count` + `act_name`/`thread_name` via JOIN), `created_at DESC` |
| `GET`    | `/brainstorm-runs/:id`        | Lauf-Detail inkl. geparstem `result` (`{ vorschlaege }`) + `act_id`/`thread_id` |
| `DELETE` | `/brainstorm-runs/:id`        | Einzel-Lauf löschen (Owner via `user_email`) |

**Validierung (Server-seitig, Pflicht):**
- `resolveFigureIds(bookId, userEmail, fig_ids)` filtert die Katalog-Figuren aufs Subset, das wirklich zu (Buch, User) gehört, und übersetzt TEXT-`fig_id` → INTEGER `figures.id` — verhindert Fremd-Verschmutzung der M:N-Tabelle. Dient auch der Strang-Bindung (`figure_id`).
- `resolveDraftFigureIds(bookId, userEmail, draft_figure_ids)` analog für Werkstatt-Figuren: filtert die INTEGER-IDs aufs (Buch, User)-Subset von `draft_figures`. Unbekannte/Fremd-IDs fallen still raus. Dient auch der Strang-Bindung (`draft_figure_id`).
- `_validChapterId(bookId, chapterId)` erzwingt, dass das Kapitel zum Buch gehört, sonst `NULL`. Dient sowohl dem Beat-`chapter_id` als auch der Strang-Kapitel-Bindung (`plot_threads.chapter_id`).
- `_validThreadId(bookId, userEmail, threadId)` (in [db/plot.js](../db/plot.js)) erzwingt, dass der Strang zu (Buch, User) gehört, sonst `NULL` — kein Fremd-Strang an einem Beat.
- Beim Beat-Anlegen/-Verschieben muss der Ziel-Akt zum selben Buch + User gehören (`ACT_MISMATCH` sonst).
- `_actFitsThread(act, threadId)` erzwingt die Hybrid-Invariante: ein Beat darf nur auf einem **geteilten** Akt (`thread_id NULL`) oder einem Akt **seines eigenen Strangs** sitzen — nie auf dem eigenen Akt eines fremden Strangs (`ACT_THREAD_MISMATCH` sonst). Geprüft bei Beat-Create + bei jedem `act_id`/`thread_id`-Move via PATCH (gegen das **resultierende** Paar).
- Limits: Titel 200, Beschreibung 4000, Akt-Name 120, Strang-Name 120, Farbe 32 Zeichen.

## KI-Jobs

Beide via Job-Queue ([routes/jobs/plot.js](../routes/jobs/plot.js), gemountet in [routes/jobs.js](../routes/jobs.js)), Prompts + Schemas in [public/js/prompts/plot.js](../public/js/prompts/plot.js) (re-exportiert über die Facade [public/js/prompts.js](../public/js/prompts.js)). **Rein planend/überwachend — nie generativ in den Text.**

| Job-Typ | Endpunkt | dedupId | Eingabe | Output |
|---------|----------|---------|---------|--------|
| `plot-brainstorm`  | `POST /jobs/plot-brainstorm`  | `${bookId}\|brainstorm\|${actId}\|${threadId\|'none'}` | `{ book_id, act_id, thread_id? }` | `{ vorschlaege:[{label, begruendung}], actId, threadId, runId }` |
| `plot-consistency` | `POST /jobs/plot-consistency` | `${bookId}`                       | `{ book_id }`         | `{ konflikte:[{beat, schwere, problem, vorschlag}], fazit, runId }` |

**Eigener System-Prompt** (`buildPlotSystemPrompt`): self-contained, **ohne** Locale-Config-Abhängigkeit (Dramaturg-/Lektor-Rolle + `_jsonOnly()`). Anders als die Figuren-Werkstatt nutzt Plot keinen `SYSTEM_FIGUREN`.

**Kontext-Loader** (beide Jobs grundieren mit Buch-Realität):
- `_figurenContext` — Katalog-Figuren-Ensemble (Name + Typ) aus `getFiguren`.
- `_werkstattFigurenContext` — Werkstatt-Figuren-Ensemble aus `listDraftFigures` (Direkt-Require von `db/draft-figures`, kein pages/chapters/books). Best-effort: bei Fehler `[]`. Fliesst als eigener Prompt-Block `FIGUREN-WERKSTATT (in Entwicklung …)`. So kennt Brainstorm geplante Figuren als Beat-Material, und Consistency beanstandet einen Beat, der eine Werkstatt-Figur referenziert, nicht als „unbekannte Figur". **Angereichert um die psychologischen Kerne der Mindmap** (Want/Need/Wound/Lie + Bogen + Konflikt) via `extractPsychologie` aus [lib/draft-mindmap-extract.js](../lib/draft-mindmap-extract.js) (liest die stabilen Default-Container-IDs `want/need/wound/lie`, `steckbrief>bogen`, `steckbrief>konflikt`); `_werkstattFigurenLines` rendert sie kompakt. So kann der Plot Beats auf den **inneren Konflikt** der Figur zuschneiden statt nur auf den Namen — Brainstorm hat dafür eine eigene Regel-Bullet.
- `_kapitelContext` — Kapitelnamen in echter Buchorganizer-Reihenfolge **über die Content-Store-Facade** (`loadOrderedBookContents`, kein Direkt-SQL auf `chapters`). Best-effort: bei Fehler `[]`.
- `_szenenContext` (nur Consistency) — extrahierte Szenen mit Kapitel + beteiligten Figuren aus `figure_scenes`/`scene_figures` (Direkt-SQL erlaubt — keine pages/chapters/books; `chapter_name` via JOIN, LIMIT 150).
- `_threadContext` (beide Jobs) — Handlungsstränge mit aufgelöster Hauptfigur + gebundenem Kapitel (`{ id, name, figur, kapitel }`). Katalog-Bindung über TEXT-`fig_id` → Name (Direkt-SQL auf `figures`), Werkstatt-Bindung über `draft_figures.id` → Name, Kapitel über `chapter_name` (aus `listThreads`-JOIN). Best-effort. Speist im Prompt den `HANDLUNGSSTRÄNGE`-Block (Name + Hauptfigur + Kapitel) + die `{Strang: …}`-Annotation pro Beat im Board-Outline + den Vererbungs-Hinweis. Im Board-Outline erbt ein Beat ohne eigenes Kapitel das Strang-Kapitel (`→ Kapitel: … (vom Strang)`); die Strang-Hauptfigur gilt laut Regel-Hinweis implizit als beteiligt.

**Brainstorm:** schlägt 3–7 prägnante Beats (3–10 Wörter Label + 1-Satz-Begründung) für **einen** Akt vor, passend zum bisherigen Board + Buchkontext + Katalog- + Werkstatt-Figuren. **Zell-granular bei Strang:** mit `thread_id` grundiert der Prompt einen `ZIEL-STRANG`-Block (Strang-Name + gebundene Hauptfigur → „Beats sollen diesen Erzählstrang vorantreiben"), und die „NICHT wiederholen"-Liste wird auf die Zelle (Akt × Strang) gefiltert. Ohne `thread_id` akt-weit wie gehabt (Abwärtskompat für ungethreadete Boards). Der Job gibt `threadId` im Result zurück, damit das Frontend den Vorschlag in der richtigen Zelle anwendet. **Jeder Lauf mit ≥1 Vorschlag wird in `plot_brainstorm_runs` historisiert** (best-effort, `runId` im completeJob-Payload) — so kann der User frühere Vorschläge später wieder ansehen und anwenden.

**Consistency:** prüft den geplanten Plot in sich + gegen die Buchrealität — Beats mit `im_buch` ohne Szenen-Entsprechung, geschriebene-aber-noch-`geplant`-Drift, Chronologie-Brüche (Beat-Reihenfolge vs. verknüpfte Kapitel), Logiklücken, fehlende Wendepunkte, verworfene-aber-noch-präsente Inhalte. **Mit Strängen** zusätzlich: vollständiger Bogen je Strang (Setup/Eskalation/Auflösung, besonders je Hauptfigur), Strang-Balance / POV-Lücken, Verweben der Stränge. Werkstatt-Figuren gelten als legitime, geplante Figuren (kein „nicht im Buch"-Befund allein deswegen). Leeres `konflikte`-Array + bestätigendes `fazit` bei Stimmigkeit. **Jeder Lauf wird in `plot_consistency_runs` historisiert** (best-effort beim Job-Complete; DB-Fehler verschluckt das Resultat nicht) und der frisch persistierte `runId` im completeJob-Payload zurückgegeben — so kann der User frühere Prüfungen später nochmal ansehen.

**Severity-Skala** `kritisch|stark|mittel|schwach|niedrig` (Schema-enforced enum `PLOT_SEVERITY_ENUM`), kompatibel zu `.severity-tag--*` aus [DESIGN.md](../DESIGN.md) — gleiche Skala wie die [Figuren-Werkstatt](figur-werkstatt.md).

**Job-Labels:** `job.label.plotBrainstorm` (`{ akt }`-Param) / `job.label.plotConsistency`. Dedup via `findActiveJobId(type, entityKey, userEmail)`.

## Frontend-Card

Sub-Komponente `plotCard` ([public/js/cards/plot-card.js](../public/js/cards/plot-card.js)), Methoden in [public/js/book/plot.js](../public/js/book/plot.js) (`plotMethods`). Buchebenen-Karte, exklusiv (`EXCLUSIVE_CARDS`-Eintrag in [feature-registry.js](../public/js/cards/feature-registry.js), `flag: showPlotCard`, `toggle: togglePlotCard`, `onReclick: refresh`).

**Lifecycle:** `setupCardLifecycle({ name:'plot', showFlag:'showPlotCard', timerKeys:['_brainstormPollTimer','_consistencyPollTimer'], onShow: loadBoard, onBookChanged: reset+reload, onViewReset: resetPlot, onCardRefresh: loadBoard })`.

**Memo:** genau ein `_memo(key, deps[], fn)`-Helper (Array-Deps shallow `===`); `beatsForAct`/`beatsForCell`/`filteredBeatsForCell`/`threadLanes`/`boardStats`/`actStats`/`tensionCurve`/`visibleBeatsForAct` memoized, `this._memos = {}` bei jeder Daten-Mutation/`loadBoard`.

**Akt-Farbe:** jeder Akt trägt eine optionale Farb-Identität (`plot_acts.farbe` = Palette-Key aus `ACT_PALETTE`, referenziert die theme-aware `--palette-*`-Tokens). Swatch im Spaltenkopf → Palette-Popover (`setActColor` → `PATCH /plot/acts/:id { farbe }`, `null` setzt zurück). `actAccent(act)` baut den CSS-Wert (`var(--palette-<key>)` oder Karten-Akzent-Fallback, Whitelist gegen Injection); das Markup bindet ihn als `--col-accent`-Custom-Prop pro Spalte. Spalten-Header-Border, Dropzone, Beat-Intensität, Intensitäts-Editor und die Spannungspunkte erben diese Farbe.

**Status-Verteilung:** board-weiter segmentierter Balken (`.plot-dist-bar` + Legende) aus `boardStats().by` — drei Segmente `geplant`/`im_buch` (nur aktive Beats) + `verworfen` (Flag-Achse, `distStatusList()`) + Mini-Variante pro Spaltenkopf aus `actStats(actId).by`. Breite via `flex-grow: var(--seg-grow)` (Count, kein Width-String).

**Spannungsbogen:** klappbares Inline-SVG-Band (`tensionCurve()`): Beats mit Intensität 1–5 in Board-Lesereihenfolge (Akt-Position → `sort_order`) als Kurve; verworfene Beats zählen nicht. Punkte (`.plot-tension-dot`, Akt-Farbe) öffnen per Klick den Beat-Edit; Polyline via `preserveAspectRatio="none"` + `vector-effect="non-scaling-stroke"`. Sichtbar ab ≥2 Punkten. Intensität wird im Beat-Edit über eine 1–5-Stufenwahl gesetzt (`setBeatDraftIntensitaet`, erneuter Klick = zurücksetzen) und in der Beat-Kopfzeile als Signal-Meter angezeigt.

**Verworfen-Collapse:** verworfene Beats sind pro Spalte einklappbar (`visibleBeatsForAct` vs. `filteredBeatsForAct`, `+N verworfen`-Toggle via `.collapsible-toggle`); Drag/Reorder operieren weiter auf der vollständigen Liste.

**Akt-Reordering** per Pfeil-Button (`moveAct(act, dir)`, a11y statt Spalten-Drag) → `PUT /plot/acts/order`.

**Beat-Reordering** per SortableJS (lazy via `loadSortable()`; Code [public/js/book/plot/dnd.js](../public/js/book/plot/dnd.js)). Der drift-anfällige SortableJS-Kern — Prototype-Patch, `revertSortable`, Präzisions-Tuning (`BASE_SORTABLE_OPTS`), x-ignore — liegt geteilt mit dem Buchorganizer in [public/js/sortable-dnd.js](../public/js/sortable-dnd.js); beim Sortable-Versionssprung dort verifizieren. Feature-Spezifisches (Grid-/Swimlane-Zellen, Griff, Reattach, Drop-Logik) bleibt im Plot-`dnd.js`. Aufbau: pro Beat-Zelle ein Container `.plot-beats[data-plot-cell]` (nur x-for-Anker + Beat-Karten), alle in der `plot-beats`-Gruppe → Drag zwischen Zellen. **Drag startet ausschliesslich am Griff** (`handle: '.plot-beat-grip'`, das `grip-vertical`-Icon oben links auf der Karte, nur im Ansichtsmodus gerendert) — so bleibt die restliche Karte voll klickbar (Titel→Edit, Tags→Figur springen) ohne Drag/Klick-Konflikt; ein bearbeiteter Beat (kein Griff) ist nie ziehbar, ein `filter` erübrigt sich. Ein **Ghost-Slot** zeigt vor dem Loslassen die Landeposition. `onBeatSortEnd` liest Ziel-`act_id`/`thread_id` (aus `data-*` der Ziel-Zelle) + die Vor-Beat-ID (nächste `.plot-beat`-Geschwisterkarte), nimmt SortableJS' physischen DOM-Move zurück (`revertSortable`, gegen Alpine-x-for-Doppelbesitz) und delegiert an die geprüfte `_dropBeat`-Mechanik (optimistisch lokal umsortiert inkl. Neu-Nummerierung Quell-+Ziel-Zelle, dann `PUT /plot/beats/order`). Bei Persist-Fehler `loadBoard()` als Rollback. Strukturelle Änderungen (Akt-/Strang-CRUD, Fork, loadBoard) binden die Container via `_scheduleReattach`/`_reattachSortables` neu (`$watch('acts'|'threads')`). Akte werden per Pfeil-Button verschoben (a11y), nicht per Drag.

**Status:** in der zugeklappten Beat-Ansicht nicht mehr als Badge gezeigt (nur die Border-Left-Farbe signalisiert ihn — Platzersparnis); gesetzt wird er über die Status-Tabs im Bearbeiten-Modus und gefiltert über die Status-Combobox in der Filter-Leiste.

**Beat-Edit:** Inline-Editor mit Status-Tabs, Kapitel-Combobox (aus `$app.tree`-Kapiteln) und **zwei** Multi-Select-Figuren-Comboboxen: Katalog (aus `$app.figuren`, **nach Kapitel gruppiert** über `opt.group` — eine Figur erscheint unter jedem Kapitel, in dem sie laut `figure_appearances` auftritt; Figuren ohne Auftritt unter „Ohne Kapitel"; Optionen via `beatFigureOptions`) und Werkstatt (aus `draftFiguren`, lokal in der Karte via `GET /draft-figures/:bookId` geladen, flach/ungruppiert via `beatWerkstattOptions`). Beide binden per `x-modelable`/`x-model` an `beatDraft.figure_ids` bzw. `beatDraft.draft_figure_ids`; die Auswahl wird darunter als entfernbare Chips (`beatFigureChips`/`beatWerkstattChips`, Remove via `toggleBeatDraftFigure`/`toggleBeatDraftWerkstattFigure`) gezeigt (Werkstatt-Chip mit gestricheltem Werkstatt-Akzent). Die Option-Builder lesen die reaktive Quelle über `window.__app.*` (Katalog) bzw. berühren `draftFiguren` inline im `x-effect` (Werkstatt) — sonst trackt die nested combobox-`x-effect` die Reaktivität nicht (siehe DESIGN.md „Reaktivität bei Datenquelle aus Karten-Scope"). Beat-Meta-Badges sind klickbar: Kapitel → `$app.openKapitelByName`, Katalog-Figur-Tags → `$app.openFigurById`, Werkstatt-Figur-Tags (`.tag--werkstatt`) → `$app.openWerkstattDraftById`. Filter-Leiste hat eine eigene Werkstatt-Figur-Combobox (`catalogFilter('werkstattFigur')`), sichtbar nur wenn Beats Werkstatt-Figuren verknüpfen, eine **Status-Combobox** (`catalogFilter('status')`, Optionen aus `statusList()`, `plotFilters.status`) und ein **Volltext-Suchfeld** (`plotFilters.text`, debounced, kein Spellcheck — Filter-Feld-Ausnahme) das Titel + Beschreibung der Beats matcht. Alle Filter wirken nur fürs Rendering (`filteredBeatsForAct`/`-Cell`); `beatsForAct`/`-Cell` bleiben ungefiltert für DnD + Order-Persistenz.

**Kapitel-Coverage** (`plotCoverage`/`plotCoverageRelevant`, lokales Aggregat — **kein** KI-Job): klappbares Panel unter der Filter-Leiste, sichtbar sobald geplant wird und es etwas zu melden gibt. Listet Buch-Kapitel ohne Handlungspunkt (klickbar → `$app.openKapitelByName`) und zählt Beats ohne Kapitel. Match über `chapter_name` (Anzeige-Join) gegen die `$app.tree`-Kapitel.

**Brainstorm-UI:** im flachen Board pro Akt ein Button, im Grid ein kompakter `#zap`-Icon-Button **pro Zelle** (nur echte Stränge — `runBrainstorm(act, thread)`) → Job-Poll (`startPoll` aus [job-helpers.js](../public/js/cards/job-helpers.js)) → Vorschlags-Panel in Akt bzw. Zelle (Match über `brainstormActId` + `brainstormThreadId`). `applyBrainstorm(idx)` legt aus dem Vorschlag direkt einen Beat an (`label`→`titel`, `begruendung`→`beschreibung`, `thread_id` aus `brainstormResult.threadId`) und entfernt ihn aus der Liste.

**Brainstorm-Historie:** persistierte Läufe (`plot_brainstorm_runs`) als klappbarer Verlauf unterhalb der Konsistenz-Historie (`history-bar`-Pattern, `created_at DESC`, Akt-/Strang-/Vorschlag-Anzahl- + Modell-Tag). `loadBrainstormRuns()` lädt die Liste mit `loadBoard()`. `openBrainstormRun(run)` holt das Detail lazy und legt `{ vorschlaege }` zurück ins **bestehende** Inline-Vorschlags-Panel des zugehörigen Akts/der Zelle (`brainstormResult` + `brainstormActId`/`brainstormThreadId` + `selectedBrainstormRunId`) — genau wie ein frischer Lauf, sodass „Übernehmen" unverändert greift; `_scrollToBrainstormPanel()` holt das eine sichtbare Panel ins Bild. Ein Lauf mit **gelöschtem Akt** (`act_id === null`) ist nicht reaktivierbar (keine Ziel-Spalte) → Eintrag gedämpft, nicht klickbar. `deleteBrainstormRun(runId)` löscht einen Eintrag (Confirm). Spiegelt das Konsistenz-Run-Pattern, nur zusätzlich akt-/strang-skopiert.

**Consistency-UI:** Button im Card-Header (deaktiviert bei leerem Board) → Panel mit `fazit` + aufklappbarer Konflikt-Liste (Severity-Tag + betroffener Beat + Vorschlag). **Geschlossene Triage-Schleife (Befund ↔ Beat):** Solange ein Lauf angezeigt wird (`consistencyResult`), trägt jeder im Lauf benannte Beat ein farbiges Warn-Badge (`.plot-beat-warn`, höchste Schwere via `beatTopSeverity`, Tooltip = die Probleme); Klick (`focusKonfliktForBeat`) klappt den ersten zugehörigen Befund auf und scrollt das Consistency-Panel ins Bild. Umgekehrt ist der Beat-Name in der Konflikt-Liste klickbar (`gotoKonfliktBeat` → `scrollToBeat`: scrollt den Beat zentriert ins Bild + kurze `.plot-beat--flash`-Hervorhebung). Abgleich rein über den Beat-**Titel** (`_konfliktIndex`/`beatKonflikte`, normalisiert) — gleiche Vertragsbasis wie der Job, der den Beat nur per Titel-String referenziert; nach Beat-Umbenennung matcht der alte Lauf den Beat nicht mehr. Beide Pfade sind `$root`-skopiert (auch im Vollbild korrekt).

**Konsistenz-Historie:** persistierte Läufe (`plot_consistency_runs`) als klappbarer Verlauf unter dem Consistency-Panel (`history-bar`-Pattern, `created_at DESC`, Konflikt-Anzahl- + Modell-Tag). `loadConsistencyRuns()` lädt die Liste mit `loadBoard()` (best-effort, eigenständig vom Board). `openConsistencyRun(runId)` holt das Detail lazy und legt `{ konflikte, fazit }` ins **bestehende** Consistency-Panel (`consistencyResult` + `selectedRunId`) — genau wie ein Live-Lauf; erneuter Klick auf den aktiven Eintrag schliesst es. Ein frisch abgeschlossener Lauf markiert sich selbst (`selectedRunId = job.result.runId`) und triggert ein Reload der Liste. `deleteConsistencyRun(runId)` löscht einen Eintrag (Confirm). Spiegelt das [Figuren-Werkstatt](figur-werkstatt.md#run-historie)-Run-Pattern, nur buch- statt draft-skopiert und ohne Live-Job-Reattach.

**Vollbild:** Icon-Button (`maximize-2`/`minimize-2`) im Card-Header → `togglePlotFullscreen()` setzt die ganze Karte (`$root` = `.card--plot`) per Native-Fullscreen-API auf den Viewport, damit das horizontal scrollende Board mehr Platz für alle Akte hat. Status `plotFullscreen` wird vom `fullscreenchange`-Listener (`attachFullscreenSync` aus [public/js/fullscreen.js](../public/js/fullscreen.js), abgemeldet via Lifecycle-`signal`) gespiegelt; CSS-Regel `.card--plot:fullscreen` in [plot/board.css](../public/css/book/plot/board.css). Esc beendet (Browser-nativ).

## Swimlanes (Akte × Stränge)

**Render-Weiche:** beide Boards liegen als Geschwister im Card-Root, je `x-show`. [plot-board-flat.html](../public/partials/plot-board-flat.html) (`x-show="(acts.length || addingAct) && !threads.length"`) ist das **unveränderte** flache Akt-Board; [plot-board-grid.html](../public/partials/plot-board-grid.html) (`x-show="threads.length"`) das Raster. [plot.html](../public/partials/plot.html) ist nur der Rahmen (Header/Progress/Consistency/Tension/Filter) + drei nested-Partial-Placeholder (`partial-plot-thread-bar`, `partial-plot-board-flat`, `partial-plot-board-grid`), gefüllt via `_loadPartials`-Cascade.

**Geteiltes Beat-Markup (SSoT):** die Beat-Karte (Ansicht + Bearbeiten) lebt **einmal** in [plot-beat-cell.html](../public/partials/plot-beat-cell.html) und wird in beide Boards per **string-seitigem Fragment-Include** `<!-- @include plot-beat-cell -->` eingebettet ([app-ui.js](../public/js/app/app-ui.js)#`_resolveIncludes`, vor `innerHTML`/`Alpine.initTree`). Anders als der DOM-Placeholder-Mechanismus (`<div id="partial-X">`) greift das auch **innerhalb** von `<template>`/`x-for` (querySelector steigt nicht in Template-Content ab). Das Fragment erwartet im Scope die `beat`-Loop-Variable + den plotCard-State; der umschliessende `.plot-beat`-Wrapper inkl. DnD-Handler bleibt pro Board-Datei (flach vs. Zelle unterscheiden sich). **Neue Beat-Felder → nur in `plot-beat-cell.html` pflegen** (kein Doppel-Edit mehr).

**Zeilen (Hybrid-fähig):** Das Grid rendert eine **flache Liste von Zeilen-Deskriptoren** aus `gridRows()` (eine `x-for`-Schleife, je Deskriptor ein Header- ODER Lane-Block; `display:contents`-Wrapper `.plot-swim-rowgroup`, damit Zeilen direkte Flex-Kinder von `.plot-swim` bleiben). Aufbau: (1) **geteilte Region** — ein Header über `sharedActs()` + alle Lanes ohne eigene Akte (Stränge **und** „ohne Strang"); ihre Spalten richten sich aus. (2) Pro Strang **mit eigener Aktstruktur** (`threadHasOwnActs`) ein **eigener** Header über `actsForThread(id)` + seine Lane darunter — eigene Spaltenzahl, bewusst nicht ausgerichtet. Da das Layout per Flexbox arbeitet (kein CSS-Grid mit fixem `--act-count`), tragen Zeilen unterschiedlicher Spaltenzahl problemlos nebeneinander. Cells weiterhin aus `filteredBeatsForCell(actId, laneId)`. Weil `gridRows` die Beat-Zell-Markup-Stelle EINMAL führt (Lane-Deskriptor mit `row.acts`), bleibt trotz Hybrid nur **eine** Zell-Kopie im Grid-File (+ die Flat-Kopie). Horizontaler Scroll via `min-width: max-content`. CSS: [plot/swimlane.css](../public/css/book/plot/swimlane.css).

**Eigene Aktstruktur (Fork-Toggle):** im Lane-Label jedes Strangs ein `git-fork`-Icon-Button → `forkThreadActs(thread)` (noch geteilt) bzw. `unforkThreadActs(thread)` (schon eigen). Beide bestätigen via `appConfirm`, rufen `POST`/`DELETE /plot/threads/:id/fork-acts` und laden danach das Board neu (`loadBoard` — der act_id-Remap betrifft viele Beats, lokales Spiegeln wäre fehleranfällig). Scoped **Akt hinzufügen** pro Header (`addingActScope`: `false`=aus, `null`=geteilt, `<threadId>`=strang-eigen) → `addActScoped`. `moveAct` reordert **innerhalb des Scopes** (`act.thread_id`), der andere Scope bleibt unberührt.

**Strang-Anlegen:** [plot-thread-bar.html](../public/partials/plot-thread-bar.html) („+ Strang", sichtbar sobald Akte existieren) — der Opt-in-Einstieg. **Strang-Management** (umbenennen/färben/verschieben/binden/löschen) lebt in den Lane-Labels des Grids: `addThread`/`saveEditThread`/`setThreadColor`/`moveThread`/`deleteThread`. Figuren-Bindung exklusiv (Katalog **oder** Werkstatt) über zwei Comboboxen im Lane-Edit; `threadDraft.figure_id` führt die TEXT-`fig_id`, `threadDraft.draft_figure_id` die INTEGER-`draft_figures.id`. Kapitel-Bindung über eine dritte Combobox (`threadDraft.chapter_id`, Optionen aus `$app.tree`-Kapiteln). `deleteThread` spiegelt den Server-SET-NULL lokal (Beats fallen in die „ohne Strang"-Lane, werden nicht gelöscht).

**Live-Vererbung im Grid:** `inheritedFigureForBeat(beat)` / `inheritedChapterForBeat(beat)` (in [public/js/book/plot.js](../public/js/book/plot.js)) liefern die vom Strang geerbten Werte für die Beat-Anzeige — geerbte Figur nur, wenn nicht schon explizit am Beat; geerbtes Kapitel nur, wenn der Beat kein eigenes `chapter_id` hat. Gerendert als gedämpfte, gepunktete `tag--inherited`/`kapitel-badge--inherited`-Badges (`plot/board.css`) neben den expliziten Tags. Lane-Label zeigt die gebundene Figur (`threadFigureLabel`) **und** das gebundene Kapitel (`lane.thread.chapter_name`). Rein Anzeige — nichts davon wird je auf den Beat geschrieben.

**2D-DnD:** flaches Board und Grid nutzen dieselbe SortableJS-Anbindung — jede Beat-Zelle (`.plot-beats[data-plot-cell]`) trägt `data-act-id` + (im Grid) `data-thread-id` (leer = „ohne Strang"). `onBeatSortEnd` liest beide aus der Ziel-Zelle und ruft `_dropBeat(actId, threadId, beforeBeatId)` — setzt `act_id` + `thread_id`, nummeriert Ziel- + Quell-**Zelle** neu, persistiert via `_persistCells` → `PUT /plot/beats/order` mit `{actId, threadId, beatIds}`-Gruppen. Fehler → `loadBoard()`-Rollback. **Leere Grid-Zellen als Drop-Ziel:** im Ruhezustand kollabieren leere Zellen auf `min-height: 0` (kein Leerraum). Während eines Drags setzt `dnd.js` die Klasse `plot-dnd-active` + die Custom-Prop `--plot-drag-h` (Höhe der gezogenen Karte) auf `<body>`; die Swimlane-CSS bläht dann **jede** Zelle auf diese Höhe auf — bewusst ohne `:not(:has())`, damit die Zielzelle nicht auf Ghost-Höhe schrumpft, sobald der Ghost eindringt (sonst fällt der Cursor aus der Trefferzone → Beat springt beim Loslassen zurück). Nach dem Drag kehren die Zellen zur kompakten Höhe zurück.

**Spannungsbogen:** `tensionCurve()` liefert weiterhin die globale `points`/`polyline`/`count` (Board ohne Stränge) **plus** `series[]` (pro Strang eine farbige Polyline). Template rendert `series` wenn Stränge existieren, sonst die globale Kurve.

## Cross-Feature: Figuren-Werkstatt

Plot und [Figuren-Werkstatt](figur-werkstatt.md) befruchten sich gegenseitig — beide rein planend, nie generativ:

- **Psychologie im KI-Kontext** (s.o.): die Werkstatt-Mindmaps (Want/Need/Wound/Lie + Bogen + Konflikt) fliessen in Brainstorm + Consistency, damit Beats den inneren Konflikt der Figur bedienen.
- **`figurePlotUsage(bookId, userEmail, { draftFigureId, sourceFigureId })`** ([db/plot.js](../db/plot.js)) — Beats, an denen eine (Werkstatt-)Figur beteiligt ist (direkt verlinkt **oder** als Strang-Hauptfigur, Live-Vererbung) + die gebundenen Stränge. Lesefertig angereichert (Akt, Strang, effektives Kapitel, Status, Intensität). Speist die Werkstatt-Jobs (umgekehrte Richtung: „Figurenbogen ↔ geplante Handlung") **und** die `/figure-usage`-Route.
- **Figur-Filter mit Vererbung:** `_beatMatchesFilter` (Katalog **und** Werkstatt) trifft auch Beats, deren Strang die Figur als Hauptfigur bindet (`_beatInvolvesCatalog`/`_beatInvolvesDraft`) — dieselbe Beat-Menge wie das Badge. `applyDraftFigureFilter(draftId)` (Event `plot:filter-draft-figure`) ist der Landepunkt der Navigation Werkstatt → Plot.
- **Figuren-Coverage** (`figureCoverage`/`figureCoverageRelevant` in [derived.js](../public/js/book/plot/derived.js), lokales Aggregat, kein KI-Job): klappbares Panel unter der Kapitel-Coverage. Zwei Lücken — *ausgearbeitete Werkstatt-Figur ohne jeden Beat* (`developedNoPlot`) und *im Plot beteiligte Figur ohne ausgearbeiteten Bogen/Subtext* (`inPlotNoArc`). `_draftHasArc` ist das Client-Pendant zu `extractPsychologie` (gleiche stabile Container-IDs); Chips → `openWerkstattDraftById`.
- **Spannungsbogen-Figur-Fokus** (`tensionFigurOptions`/`beatHasTensionFigur`/`tensionFocusLine`): Combobox im Tension-Panel hebt die Beats einer Figur (Katalog `c:<figId>` / Werkstatt `w:<draftId>`, inkl. Strang-Vererbung) auf der Kurve hervor (Rest gedämpft); im strang-losen Modus zusätzlich eine eigene Fokus-Polyline (die „emotionale Reise" der Figur). State `tensionFocusFigur`.

## Buch-Chat-Integration

Der agentische Buch-Chat liest das Board read-only über das Tool `get_plot_board` ([routes/jobs/book-chat-tools/tools-plot.js](../routes/jobs/book-chat-tools/tools-plot.js), Schema in [public/js/prompts/chat.js#BOOK_CHAT_TOOLS](../public/js/prompts/chat.js)) — siehe [buchchat-tools.md](buchchat-tools.md). Liefert Akte → Beats (Status, Zielkapitel, Strang, beteiligte Katalog-/Werkstatt-Figuren auf Namen aufgelöst) als kompakten Snapshot, damit der Chat „geplant vs. schon im Buch", Chronologie-Fragen, Strang-/POV-Verläufe und Plot-Stimmigkeit beantworten kann. Top-Level-Feld `threads` (nur wenn vorhanden) listet die Stränge mit gebundener Hauptfigur + optionalem `kapitel`; jeder Beat trägt `thread` (Strang-Name oder `null`) sowie — wo zutreffend — die geerbten Felder `geerbte_figur` (Strang-Hauptfigur, sofern nicht schon explizit) und `geerbtes_kapitel` (Strang-Kapitel, sofern der Beat kein eigenes hat). Akte tragen bei Hybrid-Aktstruktur `eigener_akt_von_strang` (sonst geteilt). Optionaler `status`-/`act_id`-Filter; `status_counts` zeigt immer die Gesamtverteilung. Pro Buch + User gescoped (`WHERE book_id = ? AND user_email = ?`, identisch zur CRUD-Skopierung). Rein lesend — der Chat plant nichts, er liest die Planung des Users; die beiden planenden KI-Jobs (Brainstorm/Consistency) bleiben der Karte vorbehalten.

## Pflicht-Invarianten

- **Nie generativ in den Text** — beide KI-Jobs liefern nur Struktur-Stichpunkte (Beats) bzw. Befunde, nie Prosa. Der System-Prompt erzwingt das explizit; bleibt bei Änderungen erhalten (App-Philosophie: KI rückwärtsgewandt/weltaufbauend, nie in den Buchtext schreibend).
- **`truncated` vor `parseJSON` prüfen + Pflichtfeld-Check** nach jedem `aiCall` (`vorschlaege`/`konflikte`/`fazit`) — sonst Partial-Daten. Bereits in [routes/jobs/plot.js](../routes/jobs/plot.js) umgesetzt.
- **Scoping `WHERE book_id = ? AND user_email = ?`** in jedem DB-Pfad — Plot (Akte, Stränge, Beats) ist pro Buch **und** User isoliert (kein geteilter Katalog).
- **Figuren-/Kapitel-/Strang-Refs validieren** (`resolveFigureIds`/`resolveDraftFigureIds`/`_validChapterId`/`_validThreadId`) vor jedem Write — keine Fremd-Verweise in den M:N-Tabellen, in `chapter_id`, `thread_id` oder der Strang-Figuren-Bindung. Katalog- und Werkstatt-Figuren sind getrennte Brücken (`fig_ids` = TEXT-`fig_id`, `draft_fig_ids` = INTEGER `draft_figures.id`); Frontend hält sie in `beatDraft.figure_ids` vs. `beatDraft.draft_figure_ids` getrennt.
- **Null Stränge = flaches Board (byte-gleich)** — das flache Akt-Board ([plot-board-flat.html](../public/partials/plot-board-flat.html)) bleibt unverändert; die Swimlanes sind opt-in (`thread_id`/`plot_threads` rein additiv). `sort_order` ist **pro Zelle** (`act_id` + `thread_id`) lückenlos.
- **Strang löschen ≠ Beats löschen** — `plot_beats.thread_id` ist `SET NULL`; gelöschte Stränge entkoppeln nur, die Beats fallen in die „ohne Strang"-Lane. **Auch bei eigener Aktstruktur:** `deleteThread` hängt die Beats strang-eigener Akte VOR dem Löschen auf die geteilten Akte um (`_landThreadBeatsOnSharedActs`), bevor `plot_acts.thread_id`-CASCADE die eigenen Akte entfernt — sonst würden die Beats mit-kaskadieren. Gibt es keine geteilten Akte, werden die eigenen stattdessen zu geteilten **befördert** (kein Beat-Verlust).
- **Hybrid-Akt-Integrität** — ein Beat sitzt nur auf einem **geteilten** Akt (`thread_id NULL`) oder einem Akt **seines eigenen Strangs**; `_actFitsThread` deckelt Create + Move (`ACT_THREAD_MISMATCH`). Akt-`position` ist **pro Scope** (geteilt bzw. je Strang) lückenlos — `moveAct`/`reorderActs` sortieren nur innerhalb eines Scopes. „Eigene Akte" ist **abgeleitet** (Existenz strang-eigener Akte), nie als Flag persistiert.
- **Vererbung ist live, nie persistiert** — die vom Strang geerbte Hauptfigur/Kapitel werden NIE auf den Beat geschrieben (keine `plot_beat_figures`-Einträge, kein `plot_beats.chapter_id`). Sie entstehen erst zur Lese-/Render-/Prompt-Zeit aus der aktuellen Strang-Bindung. Beat-Vorrang: eigene Beat-Figuren sind additiv, eigenes Beat-`chapter_id` überschreibt das Strang-Kapitel. So folgt der Beat automatisch, wenn die Strang-Bindung wechselt.
- **Kapitel-Kontext nur über die Content-Store-Facade** (`loadOrderedBookContents`), kein Direkt-SQL auf `chapters`/`pages`/`books`.
