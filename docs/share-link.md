# Share-Link (Seite/Kapitel/Buch public teilen)

Opaker Token pro Seite, Kapitel **oder** ganzem Buch. Reader sehen Read-Only-View ohne Login, können (optional) kommentieren — **allgemein** am Seitenende **oder verankert** an einer markierten Textstelle (Beta-Leser-Feedback). Verankerte + allgemeine Kommentare bilden **Threads**, auf die Reader und Owner antworten (bidirektional). Owner sieht die Links in der „Geteilte Links"-Karte (Unread-Tracking); die Kommentare selbst (verankert **und** allgemein) liest und bearbeitet er in der Kommentar-Leiste der jeweiligen Editor-Ansicht — „Kommentare anzeigen" wechselt dorthin (Seiten-Share → Notebook-Leseansicht, Buch-/Kapitel-Share → Bucheditor).

## Architektur in einem Bild

```
Browser (anon)       → GET  /share/:token          → SSR-HTML
Browser (anon)       → GET  /share/:token/threads   → JSON (Threads, no-store)
Browser (anon)       → POST /share/:token/comment   → Redirect 303 oder JSON (+ optional Anker/parent_id)
Browser (anon)       → PATCH/DELETE /share/:token/comment/:id → JSON (eigenen Kommentar erledigt/löschen, via reader_token)
Browser (anon)       → POST /share/:token/tts        → Audio-Bytes (Vorlesen, token-skopiert, no-store)
Browser (auth Owner) → /share/api/links*            → JSON-CRUD (+ POST .../comments Reply, PATCH .../comments/:id/resolve)
Browser (auth Owner) → /share/api/book-comments/:id  → JSON (alle Threads des Buchs; Kommentar-Leiste der Leseansicht)
```

`/share/:token` + `/share/:token/comment` sind die **einzigen** Public-Routen im App-Body. Mount-Reihenfolge in [server.js](../server.js): Share-Router **vor** dem Session-Guard montiert; gleichzeitig steht `/share/api/` in `API_PREFIXES`, damit Owner-Calls 401-JSON statt HTML-Redirect bekommen. Reihenfolge nicht durcheinanderbringen, sonst landet entweder Reader im Login oder Owner-API liefert HTML.

## Datenmodell (Migration 145)

Zwei Tabellen, beide via FK ans bestehende Schema gehängt:

- **`share_links`** — Token-PK, `kind IN ('page','chapter','book')` (`'book'` ab Migration 213), FKs auf `books`/`pages`/`chapters`/`app_users` (alle `ON DELETE CASCADE`), `intro` (Plaintext, max 2000 Zeichen), `expires_at`/`revoked_at`/`view_count`/`owner_last_seen_at`, `show_toc` (0|1, ab Migration 215 — optionales Inhaltsverzeichnis im Reader, nur bei `kind != 'page'` wirksam). CHECK erzwingt je nach `kind` genau das passende Ref-Set (Sentinel-frei): `page` → `page_id` gesetzt, `chapter` → `chapter_id` gesetzt, `book` → **beide** NULL (Buch via `book_id`).
- **`share_comments`** — `share_token` FK auf `share_links(token)` CASCADE, `reader_name` (optional, max 80), `body` (max 4000), `ip_hash` (SHA-256 mit Server-Salt, 16 Hex-Slice — GDPR: kein Klartext-IP). **Threading + Anker (Migration 200):** `parent_id` (Self-FK CASCADE, NULL=Root, sonst Reply — Threads eine Ebene tief, Replies erben den Anker des Roots), `anchor_bid`/`anchor_quote`/`anchor_start`/`anchor_end` (verankert an einem Block via `data-bid`, NULL=allgemeine Anmerkung), `author_email` (FK app_users SET NULL — gesetzt bei Owner-Antwort; Identitäts-Quelle ist im Code exklusiv: entweder `author_email` ODER `reader_*`), `resolved_at` (Owner-Resolve, nur Root), `reader_token` (opaker Per-Browser-Token aus localStorage für Self-Erkennung — Basis für Reader-Self-Service: eigene Kommentare erledigt-markieren/löschen). Index auf `parent_id` + `author_email`.

## Verankerung an `data-bid`

Jede verankerte Anmerkung haftet an einem Block via dessen stabiler `data-bid`-ID ([lib/html-clean.js](../lib/html-clean.js)#ensureBlockIds, bereits im persistierten Page-HTML) + dem markierten Quote-Text. Der Inhalt ist **live** (Autor editiert weiter), darum ist der Quote der Robustheits-Anker: beim Rendern wird re-verankert (Block per `data-bid` finden → `anchor_quote` darin suchen, Offsets nur als Hinweis). Findet sich der Quote nicht mehr (Block gelöscht/Text geändert), bleibt der Thread gelistet, aber **ohne** Inline-Highlight (Markierung „Stelle geändert") — nie wird die falsche Stelle markiert.

## Threads (Reader + Owner)

- **Public-Endpunkte:** `GET /share/:token/threads?rt=<reader_token>` (no-store, JSON, serverseitig leser-sicher serialisiert — nie `author_email`/`reader_token`/`ip_hash`; `mine` gegen `rt` berechnet). `POST /share/:token/comment` nimmt zusätzlich optional `parent_id`, `anchor_bid`/`anchor_quote`/`anchor_start`/`anchor_end`, `reader_token` (alle defensiv validiert: bid-Regex, Offset-Ints, Reply nur auf Root **dieses** Tokens).
- **Reader-Self-Service (eigene Kommentare):** `PATCH /share/:token/comment/:id/resolve` `{ reader_token, resolved }` markiert einen **eigenen** Root-Thread als erledigt / öffnet ihn wieder (teilt sich `resolved_at` mit dem Owner-Resolve), `DELETE /share/:token/comment/:id` `{ reader_token }` löscht einen **eigenen** Beitrag. Self-Identität ausschliesslich über `reader_token` + `author_email IS NULL` (kein Auth) — nie wird ein Owner-/Fremd-Beitrag getroffen. **Hart-Löschen nur ohne Antworten:** hat der Autor geantwortet, liefert die Route `409 HAS_REPLIES` (sonst risse die CASCADE die Owner-Antwort still mit) — das Frontend blendet den Lösch-Button dann aus und bietet nur „Erledigt" an.
- **Owner-API:** `POST /share/api/links/:token/comments` `{ parent_id, body }` (Owner-Antwort, `author_email`=Session), `PATCH /share/api/comments/:id/resolve` `{ resolved }` (Root als erledigt markieren/öffnen).
- **Reader-Frontend** ([public/js/share-reader.js](../public/js/share-reader.js)): Selektion im Artikel → schwebender „Kommentieren"-Button → Composer; Inline-Highlights via **CSS Custom Highlight API** (`::highlight(share-anchor)`, kein DOM-Eingriff am Content → kein neuer XSS-Sink); Klick auf Highlight (caret-Mapping) fokussiert den Thread; Reader-Identität (`reader_token` + Name) in localStorage. Selbstständige Widgets unter `share-reader/`: **layout** (vertikale Verankerung der Karten, Google-Docs-Modell — teilt die pure Kollisions-Mathematik [public/js/comment-card-layout.js](../public/js/comment-card-layout.js) mit den zwei SPA-Leisten) + **composer** (Selektions-Button + Anmerkungs-Overlay) neben dom/identity/menu/theme/toc/progress. Kommentar-Köpfe tragen einen Initialen-Pip (deterministische Hue pro Person) aus der pure SSoT [public/js/avatar.js](../public/js/avatar.js) — gleiche Person → gleiche Pip-Farbe in Reader und SPA (Presence-Pips + Kommentar-Leisten).
- **Zwei Kommentar-Zonen (Google-Docs-Modell).** **Verankerte** Anmerkungen schweben rechts als Margin-Rail (`.share-comments`) auf der **Höhe ihrer Textstelle** — Pendant zur Bucheditor-Leiste, hier standalone ohne Alpine. Die Positionierungs-Ebene ist `.share-comments__list`; pro Karte ein Marker-Tick (`.share-thread-marker`, echte Anker-Höhe vor Kollision). Layout-Logik ([public/js/share-reader/layout.js](../public/js/share-reader/layout.js)): Artikel + Spalte teilen den Fenster-Scroll und beginnen in derselben Grid-Zeile → Abstand „Textstelle → Spalten-Oberkante" scroll-invariant, Layout nur bei Render/Resize/Auswahl (Doppel-rAF, ResizeObserver auf Artikel + Karten), Kollisions-Auflösung greedy top-down mit Pin der aktiven Karte. Unter 1100px (Flach-Modus) fällt die Spalte auf eine statische Stapel-Liste zurück (`--comment-top`/`--marker-top` werden gelöscht, CSS stapelt). **Allgemeine** (nicht verankerte) Kommentare leben in einer abgesetzten Sektion **ganz unten** (`.share-general`) mit **eigenem Formular** für neue allgemeine Anmerkungen (das klassische SSR-`<form>`, progressive Enhancement). Beide Zonen rendern volle Threads (Reply/Resolve/Delete). **Name zentral statt pro Formular** (`setupIdentity()`): beim ersten Laden öffnet ein Namens-Modal („Dein Name", optional → „Anonym bleiben" merkt sich den Verzicht pro Tab-Session via `sessionStorage`), danach steht ein Identitäts-Chip oben rechts bei den Optionen (in den `#share-theme`-Cluster injiziert): „Als <Name> · Ändern" bzw. „Name setzen". Composer, Reply-Form **und** SSR-Form haben **kein** eigenes Namensfeld mehr — sie kommentieren mit `savedName()` (sonst anonym); Thread-Antworten tragen den Namen damit ebenfalls. SSR-Fallback ohne JS: behält das beschriftete `reader_name`-Feld (JS entfernt es), nur allgemeine Root-Anmerkungen + klassische Form. **Live-Poll (Reader-Seite):** `share-reader.js` pollt `GET /share/:token/threads` alle 10 s still — so erscheinen **Autor-Antworten ohne Reload**. `fetchThreads()` rendert nur bei echter Thread-Änderung (Signatur aus id/parent/resolved/body-Länge → kein Scroll-Reset bei jedem Tick); der Tick pausiert, solange der Leser interagiert (Composer offen, Text markiert, Eingabefeld fokussiert, halb getippter Beitrag) oder der Tab im Hintergrund ist (`document.hidden`). Gegenstück zum Owner-Poll, aber Reader-Richtung.
- **Owner-Karte:** listet **nur** die Links (Status, Ziel, View-/Comment-Counts, Unread-Badge) — **keine** Kommentar-Anzeige mehr in der Karte. „Kommentare anzeigen" (`showCommentsForLink`) markiert den Link als gesehen (`mark_seen=1`, Unread sofort 0) und **wechselt in die passende Editor-Ansicht**: **Seiten-Share → Notebook-Leseansicht** der Seite (`gotoPageById` + Event `comments-rail:goto` ohne `bid` → Leiste auf), **Buch-/Kapitel-Share → Bucheditor** (`toggleBookEditorCard` + Event `book-editor:goto-comment` ohne `bid` → Leiste auf). Dort liegen Threads, Reply, Resolve/Reopen und Delete — für verankerte **und** allgemeine Kommentare. Unread-Query zählt nur Reader-Kommentare (`author_email IS NULL`) neuer als `owner_last_seen_at` — eigene Antworten sind nie „neu".
- **Live-Poll:** Solange die Karte sichtbar ist, pollt sie alle 5 s still (`_quietRefresh`, kein Loading-Flicker): aktualisiert `view_count`/`comment_count`/`unread_count` aller Links in-place (kein vollständiger `x-for`-Reflow, solange das Token-Set gleich bleibt). So erscheinen neue Reviewer-Kommentare — egal ob über Buch-, Kapitel- oder Seiten-Share — binnen ~5 s als Unread-Badge beim Owner. Timer via `timerKeys: ['_pollTimer']` im Card-Lifecycle (Auto-Cleanup bei `book:changed`/`view:reset`/`destroy`); zusätzlich stoppt ein `$watch` auf `showShareLinksCard` den Poll beim Ausblenden, und `document.hidden` überspringt Polls im Hintergrund-Tab.
- **Per-Kommentar-Sprung in der Leiste:** Klick auf einen verankerten Thread in einer der beiden Editor-Leisten hebt die Stelle via `::highlight(comment-rail-anchor-active)` (Notebook) bzw. `::highlight(book-editor-comment-anchor-active)` (Bucheditor) hervor und scrollt hin. Die Re-Anchor-Logik (`locateRange`/`resolveCurrentQuote`) liegt als SSoT in [public/js/share-anchor.js](../public/js/share-anchor.js) und wird von Reader, Notebook-Leiste und Bucheditor-Leiste geteilt. Drei geteilte SSoT-Module tragen alle drei Kommentar-Oberflächen (Share-Reader, Notebook-Leseansicht, Bucheditor): **Re-Anchoring** [share-anchor.js](../public/js/share-anchor.js), **Kollisions-Geometrie** der schwebenden Karten (pure, Node-testbar) [comment-card-layout.js](../public/js/comment-card-layout.js) und **Avatar-Pips** [avatar.js](../public/js/avatar.js). Die beiden SPA-Leisten (Notebook + Bucheditor) teilen zusätzlich den Verhaltens-Kern [comment-rail-core.js](../public/js/editor/comment-rail-core.js) (Laden/Threads/Reply/Resolve/Delete/Triage-Filter) **und** die Layout-DOM-Glue [comment-rail-layout.js](../public/js/editor/comment-rail-layout.js); der Share-Reader hat seine eigene Vanilla-Glue ([share-reader/layout.js](../public/js/share-reader/layout.js)), nutzt aber dieselbe pure Geometrie.

### Kommentar-Leiste im Bucheditor

Der Bucheditor ([book-editor.md](book-editor.md)) zeigt Leser-Kommentare des **ganzen Buchs** als rechte Margin-Rail (dritte Layout-Spalte: Outline | Stream | Kommentare). Pendant zur Notebook-Read-Modus-Leiste, aber Scope = ganzer `.book-editor-stream` statt Einzelseite. Verhalten kommt aus dem geteilten Kern [public/js/editor/comment-rail-core.js](../public/js/editor/comment-rail-core.js) (SSoT mit der Notebook-Leiste; `createCommentRail(cfg)`); die vertikale Verankerung aus dem ebenfalls geteilten [public/js/editor/comment-rail-layout.js](../public/js/editor/comment-rail-layout.js) (`createCommentLayout(cfg)`, DOM-Glue um die pure `comment-card-layout.js`). [public/js/editor/book-editor-comments.js](../public/js/editor/book-editor-comments.js) liefert nur die Bucheditor-Glue (Scope, Highlight-Namen, Stream-Ready-Check) + spreadet beide Bündel und wird in `bookEditorCard` gespreadet. **Verankerte** Threads (mit „Stelle geändert"-Diff) **und** ein darunter abgesetzter Abschnitt mit **allgemeinen** (nicht-verankerten) Kommentaren — hier aller Link-Scopes des Buchs (`generalFilter: () => true`). Owner-Reply/Resolve/Delete identisch zur Notebook-Leiste; Daten aus `/share/api/book-comments/:book_id` (Zeilen tragen zusätzlich `link_kind`/`link_page_id`). Kein Eingriff in die Notebook-Leiste.

**Vertikale Verankerung (Google-Docs-Modell, geteilt mit der Notebook-Leiste über `comment-rail-layout.js` + pure `comment-card-layout.js`).** Statt einer simplen sortierten Liste schwebt jede verankerte Karte auf der **Höhe ihrer Textstelle** und scrollt mit dem Stream mit. Verankerte Threads liegen absolut positioniert (`--comment-top`) in einer Positionierungs-Ebene `.book-editor-comments__layer` innerhalb der Spalte; ein Marker-Tick im linken Spaltenrand zeigt zusätzlich die **echte** Anker-Höhe (`--marker-top`, vor Kollision). Da Stream und Spalte denselben Scroll-Container teilen und in derselben Grid-Zeile beginnen, ist der Abstand „Textstelle → Spalten-Oberkante" scroll-invariant → Layout wird nur bei Render/Resize/Edit/Auswahl neu gerechnet (`_layoutCommentCards`, Doppel-rAF), **nicht** pro Scroll. Kollisions-Auflösung: greedy top-down mit `CARD_GAP`; bei Auswahl wird die aktive Karte auf ihre exakte Anker-Höhe **gepinnt** und die übrigen darum herum verteilt. Höhenänderungen (async „Stelle geändert"-Diff, aufgeklappte Reply-Form, Stream-Reflow) triggern via gemeinsamem `ResizeObserver` (Stream + alle Karten) ein Re-Layout; loop-sicher, weil das Layout nur `top` setzt. **Pflicht-Invariante:** Karten/Anker-Geometrie nie im reaktiven State halten (Range-Objekte brechen als Alpine-Proxy) — Anker-Y pro Pass frisch via `locateRange` lokalisieren; nur die numerischen `_anchorY`/`_railTop` landen reaktiv auf den Thread-Objekten. Unter 800px (Flach-Modus) fällt die Spalte auf eine statische Stapel-Liste zurück (kein eigener Scroll-Container), das JS überspringt das Layout (`matchMedia(FLAT_BELOW)`). CSS in [public/css/editor/book/book-editor.css](../public/css/editor/book/book-editor.css).

Schreib-/Lesepfad: ausschliesslich [db/share-links.js](../db/share-links.js). Reader-Content kommt über die **Content-Store-Facade** ([lib/content-store/](../lib/content-store/)), nicht direkt aus SQL. **Buch-Share** (`kind='book'`) rendert das ganze Buch in Lesereihenfolge über [lib/load-contents.js](../lib/load-contents.js) (`scope:'book'`): Kapitel-Überschrift + Seiten-Blöcke, inkl. Sub-Kapiteln (anders als der Kapitel-MVP). Das Stream-HTML baut der **geteilte Renderer** [public/js/manuscript-render.js](../public/js/manuscript-render.js) (`renderStreamHtml(fromGroups(groups))`) — serverseitig per dynamic `import()` geladen (Muster wie [lib/prompts-loader.js](../lib/prompts-loader.js)), dieselbe SSoT wie Bucheditor + Fassungen-Reader. Markup: neutrale `.ms-chapter`/`.ms-page`/`.ms-page__title`/`.ms-page__body`-Klassen ([public/css/components/manuscript-stream.css](../public/css/components/manuscript-stream.css), Token-Brücke `--ms-*`→`--share-*` in `share.css`); Body-Typografie weiterhin via `.share-content`. **Escaping-Invariante:** Namen `escHtml`, `pd.html` verbatim (bereits html-clean-sanitisiert, trägt `data-bid`). Kapitel-Share nutzt `omitChapterHeaders` (Kapitelname steht im Seiten-Header), Seiten als `h2`. Owner-Trigger: Quick-Action in der Buch-Übersicht (`openShareLinksForBook`) + `'Ganzes Buch'`-Option im Create-Form der Karte. `locate` für verankerte Kommentare durchsucht bei `kind='book'` alle Buch-Seiten.

## Token

22 Zeichen base64url aus `crypto.randomBytes(16)` (~128 bit). UNIQUE-Constraint + 3x Retry bei Kollision. Regex-Vorvalidierung in der Reader-Route: `^[A-Za-z0-9_-]{16,32}$`.

## Reader-View (SSR ohne Alpine)

Zwei statische Templates werden beim Modulladen einmal in `routes/share.js` per `fs.readFileSync` geladen:

- [public/share.html](../public/share.html) — OK-View mit Platzhaltern `{{title}}`, `{{layout_class}}` (`share-layout--has-toc` bei aktivem TOC), `{{intro_block}}`, `{{toc_block}}`, `{{content_html}}`, `{{anchored_comments_html}}` (SSR leer → JS hydriert die schwebende Rail), `{{general_comments_html}}` (allgemeine Kommentare, untere Sektion), `{{form_block}}` (allgemeine Kommentar-Form, ebenfalls unten), i18n-Strings `{{t_*}}` inkl. `{{t_general_heading}}`, `{{lang}}`.

**Inhaltsverzeichnis (`show_toc`).** Owner-Opt-in im Create-/Edit-Form (nur Buch-/Kapitel-Shares). Aktiv → `buildTocBlock` (in [routes/share.js](../routes/share.js)) rendert eine `<aside class="share-toc">` mit Anker-Links als **linke Leiste** (Pendant zur Bucheditor-Outline): im Grid (≥1100px) sticky links neben dem Text (Layout `toc | Text | Kommentare`, Hook-Klasse `share-layout--has-toc` auf dem Layout-Container, gesetzt nur wenn der TOC-Block nicht leer ist), gestapelt (mobil) als gerahmte Box über dem Inhalt. `loadContentForLink` vergibt beim Aufbau der Sektionen laufende `id="secN"` an Kapitel-/Seiten-Überschriften und liefert ein paralleles `toc`-Array (`{ level, label, anchor }`); bei Buch-Shares Kapitel = Level 1, Seiten = Level 2, bei Kapitel-Shares Seiten = Level 1. TOC wird nur ab ≥2 Einträgen gerendert. **Reader-seitiges Ein-/Ausblenden:** `share-reader.js#setupTocToggle` setzt einen Aktions-Button (`.share-action-btn`) in die Werkzeug-Leiste oben rechts (`#share-actions` im `.share-header__tools`-Cluster, neben dem Theme-Switcher — Platz für weitere Aktionen). Klick blendet das ganze TOC-Panel aus (`.share-toc-collapsed` auf dem Layout-Container; im Grid gibt es die linke Spalte frei, der Text rückt nach); Zustand persistiert pro Browser in localStorage (`sw_share_toc_collapsed`). Progressive Enhancement — ohne JS bleibt das TOC offen.
- [public/share.gone.html](../public/share.gone.html) — 410-View für `revoked_at` bzw. `expires_at <= now`.

`fillTemplate(tpl, vars)` ersetzt `{{key}}` einfach. **Alles, was aus DB/User kommt, wird vor der Interpolation per `escHtml()` escaped** — Ausnahme: Page-HTML (Content-Store), das bereits beim Save durch [lib/html-clean.js](../lib/html-clean.js) sanitisiert wurde.

**Vorlesen (TTS / Proof-Listening).** Ist das Feature aktiv (`tts.enabled`), trägt `config_json` einen `tts: { enabled, pause }`-Block; share.html lädt zusätzlich [public/js/share-reader/tts.js](../public/js/share-reader/tts.js) als eigenes Modul (selbst-bootstrappend), das einen schwebenden Vorlese-Dock baut und den Text satzweise über `POST /share/:token/tts` (public, token-skopiert, Synthese-Kern [lib/tts-synth.js](../lib/tts-synth.js)) vorliest. Reines Lesen, keine DOM-Mutation (CSS Custom Highlight). Details: [docs/tts.md](tts.md). **Pre-Auth-Serving:** Der komplette Reader-Modulgraph muss für den anonymen Leser pre-auth erreichbar sein — `PUBLIC_ASSET_PREFIXES` enthält `/js/share-reader/`, `PUBLIC_ASSETS` die geteilten Module (`share-anchor.js`, `avatar.js`, `editor/comment-threads.js`, `scroll-fade.js`, `comment-card-layout.js`, `tts-segment.js`); sonst redirected der Auth-Guard den Modul-Request auf `/login` (nosniff → Modul verweigert, JS-Enhancement fällt still aus).

Reader-Form benutzt **klassischen `<form method="POST">` mit 303-Redirect-Fallback** + progressive Enhancement via [public/js/share-reader.js](../public/js/share-reader.js) (fetch + inline Status). Darum verteilt der POST-Handler je nach `Content-Type` (`application/json` → JSON-Response, sonst → Redirect mit `?cmt=ok|rate|empty|long|err|gone`). Heisst: View funktioniert auch ohne JS.

Sprache wird per `Accept-Language` erkannt (DE Default, EN-Prefix → EN). Server-i18n via [lib/i18n-server.js](../lib/i18n-server.js); Owner-User-Setting ist explizit **nicht** massgeblich (Reader hat kein Konto).

## Rate-Limit + Spam-Schutz

POST-Comment hat **zwei** Schichten:

1. **Honeypot** `_hp`-Input (hidden via CSS, `tabindex="-1"`) — gesetzt → 400, geloggt mit Token-Prefix.
2. **In-Memory-Bucket** in [lib/share-ratelimit.js](../lib/share-ratelimit.js): 30 Kommentare pro `(token, ip_hash)` pro 60 min (grosszügig, da Beta-Leser viele Inline-Anmerkungen pro Sitzung hinterlassen). Process-Restart resettet — bewusst, kein Cluster-Setup. IP-Hash: SHA-256 mit `SHARE_IP_SALT` (Env oder zufällig pro Prozess) + 16-Hex-Slice. Bei Hit setzt der Handler `Retry-After`-Header.

`db/share-links.js#countRecentCommentsByTokenIp` ist als DB-Pendant da, wird aktuell nicht eingesetzt — für künftige Cluster-Persistenz vorbereitet.

### Kommentar-Leiste in der Leseansicht

Beim Lesen einer Buchseite (Notebook-Editor, **Read-Modus**) erscheinen Leser-Kommentare als Margin-Rail rechts neben dem Text (Google-Docs-Stil). Eigenständige, datengetriebene Sub-Karte — **keine** exklusive/Palette-Karte: sie blendet sich allein ein, sobald die offene Seite Kommentare hat.

- **Daten:** `GET /share/api/book-comments/:book_id` (`db/share-links.js#listCommentsByOwnerBook`) liefert alle vollen Threads (Root + Antworten, inkl. `resolved_at`/`author_*`/`anchor_*` + `link_kind`/`link_page_id`) über alle Links des Owners zum Buch; jede Zeile trägt `share_token`. Optional `?mark_seen=1` → `markOwnerSeenForBook`.
- **Frontend:** [public/js/cards/editor-comments-card.js](../public/js/cards/editor-comments-card.js) (State) + [public/js/editor/comments-rail.js](../public/js/editor/comments-rail.js) (Glue) über den geteilten Verhaltens-Kern [public/js/editor/comment-rail-core.js](../public/js/editor/comment-rail-core.js) + die geteilte Layout-Glue [public/js/editor/comment-rail-layout.js](../public/js/editor/comment-rail-layout.js) (vertikale Verankerung wie im Bucheditor — diese Leiste schwebt ihre Karten ebenfalls auf Anker-Höhe, Flach-Modus <1100px) + pure [public/js/editor/comment-threads.js](../public/js/editor/comment-threads.js) (`groupThreads`, unit-getestet). Partial [public/partials/editor-comments.html](../public/partials/editor-comments.html), CSS [public/css/editor/notebook/comments-rail.css](../public/css/editor/notebook/comments-rail.css). Pro Buch einmal geladen. **Verankerte** Kommentare via `locateRange` ([share-anchor.js](../public/js/share-anchor.js)) auf die gerenderte `.page-content-view` gefiltert; sie schweben — wie im Bucheditor — kollisions-aufgelöst auf **Höhe ihrer Textstelle** (`--comment-top` in `.comment-rail__layer`, Marker mit `--marker-top` im linken Spaltenrand). **Allgemeine** (nicht-verankerte) Kommentare landen in einer abgesetzten, zentrierten Bande **ganz unten** unter dem Seitentext (`.comment-rail--general`) — hier nur die des **Page-Shares genau dieser Seite** (`generalFilter`: `link_kind==='page' && link_page_id===currentPage.id`; Buch-/Kapitel-Share-Allgemeines hat keine eindeutige Seite und lebt im Bucheditor).
- **Auswahl:** Klick auf einen Thread hebt die Textstelle hervor (`::highlight(comment-rail-anchor-active)`; alle Stellen dezent via `comment-rail-anchor`) und scrollt hin; der Thread klappt Aktionen + Antwort-Box auf. **Voll interaktiv** (antworten/erledigt/löschen) über die bestehenden Owner-Endpoints (Token aus `comment.share_token`).
- **Stelle seither geändert (Kommentar-Diff):** Hat der Autor den verankerten Text seit dem Kommentar geändert, findet `locateRange` keinen Range mehr. `resolveCurrentQuote(view, anchor)` ([share-anchor.js](../public/js/share-anchor.js)) trennt dann „andere Seite" (`gone` → übersprungen) von „Stelle geändert" (`changed` → Block via `data-bid` da, Quote weg). Solche Threads bleiben in der Leiste mit Badge `share.comments.textChanged` + einem Inline-Diff `anchor_quote` (damals) → aktueller Block-Text (jetzt), gerendert mit dem geteilten `renderInline` ([page-revision-diff.js](../public/js/page-revision-diff.js), `diffLib` lazy via `loadDiff`). Quote ist serverseitig auf 600 Zeichen gekappt → der Diff ist quote-scoped. Owner-only; der öffentliche Reader behält seinen Text-Hinweis.
- **Layout:** Grid-Split `.editor-body-wrap.comments-split` (Text | Rail), gesteuert über das Root-Flag `pageCommentRailOpen`, das die Karte spiegelt. Text und Rail teilen den **Fenster-Scroll** (kein eigener Spalten-Scroll mehr) — Voraussetzung für die scroll-invariante Verankerung. Im Edit-Modus und während des Lektorat-Splits (`checkDone`) ist die Leiste ausgeblendet; eingeklappt zeigt ein schwebender Chip die Anzahl. Unter 1100px Flach-Modus: gestapelte Liste, keine Verankerung (`matchMedia(FLAT_BELOW)`), allgemeine Bande darunter.
- **Triage-Filter (geteilt mit Bucheditor):** Status (alle/offen/erledigt, `commentFilterStatus`) + Reviewer-Combobox (`commentFilterReviewer`, gespeist aus den distinct Leser-Namen via `_railReviewerNames`). Filter im geteilten Kern (`_railSetFilter` → Recompute); ein Filter, der den selektierten Thread ausblendet, deselektiert ihn. i18n `share.comments.filter.*`.
- **Avatar-Pips (geteilt):** jeder Kommentar-Kopf trägt einen Initialen-Pip mit deterministischer Hue pro Person (`commentAvatar` → [avatar.js](../public/js/avatar.js)); Leser leiten Initialen/Hue aus dem Anzeigenamen ab, Owner-Antworten aus `author_display_name`/`author_email`. Gleiche Person → gleiche Farbe in Reader, Notebook- und Bucheditor-Leiste sowie den Presence-Pips.

## Sicherheits-Invarianten

- **Owner-Check**: jede `/share/api/*`-Route filtert via `link.owner_email === req.session.user.email`. ACL aufs Buch (`editor`+) wird beim `POST /share/api/links` per [lib/acl.js](../lib/acl.js)#`requireBookAccess` geprüft.
- **CSRF bewusst aus**: Reader-Comment-POST kommt explizit cross-origin (Link auf WhatsApp/Mail → fremde Domain → eigene Domain). Kein State-Change auf Owner-Account, Body steht sowieso im UI — Risiko akzeptiert.
- **XSS**: Page-HTML aus DB ist bereits via html-clean sanitisiert; alles andere durchläuft `escHtml`. Owner-Karte rendert Kommentar-Body via Alpine-Text-Bindung, **keinen** `x-html`-Sink.
- **GDPR**: nur IP-Hash gespeichert. Reader-Name optional, vor Render Server-Side escaped.

## Cache-Headers

Reader-Response setzt `Cache-Control: no-store` — Content ist live (Autor editiert weiter, ein Reload zeigt garantiert den neuesten Stand; neue Threads/Antworten holt zusätzlich der 10-s-Live-Poll, siehe „Threads"). Service-Worker-`SHELL_CACHE` precachet `/share` (Standalone-Shell) und `/js/share-reader.js`.

## Erweiterungs-Hinweise

- **Cluster/Mehrprozess-Setup**: Rate-Limit auf DB umstellen (Helper steht in `db/share-links.js`).
- **Page-Range statt ganzem Kapitel**: aktuell lädt Chapter-Share alle Pages der Direkt-Children über `listPages(book_id)` + Filter — Sub-Kapitel werden **nicht** mitgenommen (MVP). Erweiterung müsste die Hierarchie traversieren (siehe [docs/chapter-hierarchy.md](chapter-hierarchy.md)).
- **Mail-Notification** bei neuem Kommentar: implementiert via `notify.maybeNotifyShareComment` (Template `share-comment-owner`, [lib/mailer-templates.js](../lib/mailer-templates.js)). Owner-Mail, gedrosselt pro Link (ein Throttle-Fenster — eine Lese-Sitzung mit vielen Inline-Anmerkungen löst nicht eine Mail-Flut aus), Opt-out via `mail.notify.owner_on_share_comment === false` (Default an). Owner-eigene Antworten lösen nichts aus.
- **Snapshot-Modus** (Inhalt zum Share-Zeitpunkt einfrieren) wäre eigene Spalte `frozen_html`/`frozen_at` plus Toggle bei Create — aktuell live.
