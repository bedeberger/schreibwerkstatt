# Fassungen (Manuskript-Meilensteine)

Ganze-Buch-Schnappschüsse: der User friert den aktuellen Stand des **gesamten** Buchs per Knopfdruck als „Fassung 1/2/3…" ein, vergleicht zwei Fassungen miteinander, **öffnet** eine Fassung nur-lesend (Reader), **exportiert** sie (HTML/TXT/MD/EPUB/DOCX + Custom-PDF), **markiert** eine Fassung als veröffentlichte Auflage (Publikations-Anker + Lösch-Schutz) und kann das Buch auf eine Fassung **zurücksetzen** (Restore). Reader + Export operieren auf dem selbsttragenden `content_json` der Fassung — der aktuelle Buchstand bleibt dabei unberührt. Pro Buch skopiert, user-initiiert (+ eine Auto-Fassung beim Fertig-Markieren), kein Pruning (Gegenstück zu `page_revisions`, die seitenweise + automatisch entstehen). Route: `#book/:bookId/fassungen`.

## Datenmodell

Eine Zeile pro Fassung in `book_snapshots` ([db/book-snapshots.js](../db/book-snapshots.js), Schema in [db/migrations.js](../db/migrations.js) / [db/squashed-schema.js](../db/squashed-schema.js), ERD in [docs/erd.md](erd.md)):

- `id` (PK), `book_id` (FK → `books(book_id)` ON DELETE CASCADE)
- `seq` — fortlaufende Fassungsnummer pro Buch (1-basiert, **monoton**; Löschungen recyceln keine Nummer, damit „Fassung 3" stabil bleibt). `_nextSeq` = `MAX(seq)+1`.
- `label`, `description` — optionaler Name + Notiz (geclippt auf 120 / 1000 Zeichen).
- `content_json` (NOT NULL) — **selbsttragende** Momentaufnahme im `buildBookJson`-Format `{ book: { name, description, settings }, tree: [node…] }` mit Seiten-HTML **inline**. Spiegelt den swbook-Export ([routes/book-migration.js](../routes/book-migration.js)), nur als DB-Zeile statt ZIP. Bleibt gültig, auch wenn Seiten später gelöscht werden. Jede Page trägt ihre `srcId` (alte page_id) mit — Diff matcht darüber.
- `extras_json` — `collectExtras(bookId, { analysis, lektorat })` (reiner DB-Read, kein KI-Call). Der Rohblob wird gespeichert, aber **nicht** an den Client geliefert und beim Restore **nicht** wiederhergestellt (bewusst — ein Voll-Restore der Analyse-/Lektorat-Tabellen ist nicht Teil der Inhalts-Fassung). `GET /snapshots/:bookId/:id` liefert daraus eine **kompakte Zähl-Übersicht** (`extras_summary` via `_extrasSummary`: Figuren/Orte/Szenen/Ereignisse/Weltfakten/Kontinuitäts-Befunde/Ideen/Lektorats-Befunde, nur Blöcke >0) — der Reader zeigt sie als Publikations-Nachweis „so sah der Weltaufbau/das Lektorat zu diesem Zeitpunkt aus".
- `published_at` — `NULL` = nicht veröffentlicht; sonst ISO+Z-Zeitpunkt der Markierung als **veröffentlichte Auflage**. Publikations-Anker (welche Fassung erschienen ist, mit eingefrorener ISBN/Titelei aus `publication_json`) + **Lösch-Schutz** (Delete verlangt `?force=1`). Mehrere Fassungen dürfen gleichzeitig markiert sein (eine pro Auflage). Gesetzt via `POST /snapshots/:bookId/:id/publish` (`setPublished`), in `listSnapshots` + `GET single` ausgeliefert.
- `publication_json` — **eingefrorene, selbsttragende Kopie** von `book_publication` zum Capture-Zeitpunkt (freeze-only, nicht pro Fassung editierbar): `{ meta: {…getMeta-Textfelder inkl. epub_*…}, cover: {b64,mime}|null, authorImage: {b64,mime}|null }`. Nur gesetzt, wenn eine echte `book_publication`-Zeile existiert (`getMeta().created_at`), sonst `null` → Export fällt auf die Live-Defaults zurück. Cover/Autorfoto werden als base64 mitgeführt (Fassung ist damit auch bzgl. Titelei/Cover self-contained). Der Fassungs-Export (PDF/EPUB/Sync) nutzt diese Kopie via `snapshotPublication` ([lib/snapshot-export.js](../lib/snapshot-export.js)); Restore schreibt sie zurück. `listSnapshots` liefert `has_publication` als Flag (analog `has_extras`), der volle Blob wird nie in der Liste ausgeliefert.
- `chars` / `words` / `pages` / `chapters` — Kennzahlen, berechnet beim Erstellen aus dem inline-HTML (gleiche Normalisierung wie `page_stats` via `htmlToPlainText`). Kein retroaktives Update.
- `user_email`, `created_at` (ISO+Z).
- Index: `idx_book_snapshots_book (book_id, created_at DESC)`.

## Routen ([routes/snapshots.js](../routes/snapshots.js))

Alle book-scoped (`setContext({ book })`), ACL via `requireBookAccess`. Synchroner Pfad (reiner DB-Read/-Write, kein KI-/Netz-Call) — bewusste Ausnahme zur Job-Queue-Regel, analog zum Capture.

- `GET /snapshots/:bookId` (viewer) — Meta-Liste **ohne** `content_json`/`extras_json` (können MB groß sein), `has_extras` als Flag, DESC nach `created_at`.
- `GET /snapshots/:bookId/:id` (viewer) — Vollzeile inkl. geparstem `content` + `published_at` + `publication` (Text-Meta ohne Cover/Foto-BLOBs, für den Metadaten-Diff) + `extras_summary` (Zähl-Übersicht). Quelle für Diff, Reader **und** Publikations-Metadaten-Diff.
- `GET /snapshots/:bookId/:id/export/:fmt` (viewer) — synchroner Fassungs-Export in `html`/`txt`/`md`/`epub`/`docx`. Baut das Export-Bundle via `snapshotToBundle` ([lib/snapshot-export.js](../lib/snapshot-export.js)) aus dem `content_json` (statt Live-Buch), rendert mit demselben `FORMATS[fmt].build` wie [routes/export.js](../routes/export.js) und teilt dessen Meta-/Antwort-Helper ([lib/export-send.js](../lib/export-send.js)). `pdf` ist hier bewusst **ausgeschlossen** (`BAD_FORMAT`) — PDF läuft über den Job (Profile). Filename-Slug `…-fassung-<seq>`. Fehler: `SNAPSHOT_NOT_FOUND` (404), `CORRUPT_SNAPSHOT` (422).
- `POST /snapshots/:bookId` (editor) — „Fassung speichern". Baut Payload via `_buildSnapshotPayload` (geteilt mit dem Restore-Backup) und legt eine Zeile an. `BOOK_EMPTY` → 400.
- `POST /snapshots/:bookId/:id/restore` (editor) — **destruktiv**, setzt das Buch auf die Ziel-Fassung zurück (siehe unten).
- `POST /snapshots/:bookId/:id/publish` (editor) — Body `{ published: bool }`, markiert die Fassung als veröffentlichte Auflage (`published_at` setzen/nullen).
- `DELETE /snapshots/:bookId/:id` (editor) — veröffentlichte Fassung (`published_at` gesetzt) ist geschützt: ohne `?force=1` → `409 SNAPSHOT_PUBLISHED` (Frontend zeigt eine stärkere Bestätigung, dann erneut mit `force`).

Zusätzlich exportiert das Modul `captureSnapshot(bookId, req, { label, description, dedup, userEmail })` — wiederverwendbarer, best-effort (wirft nie) Capture-Einstieg für Auto-Sicherungen aus anderen Routen. `dedup:true` überspringt, wenn die jüngste Fassung denselben (chars/pages/chapters)-Stand hat. Genutzt von [routes/booksettings.js](../routes/booksettings.js): beim Fertig-Markieren des Buchs (`is_finished` 0→1) wird fire-and-forget eine Fassung `__i18n:snapshots.autoFinishLabel__` festgehalten (Publikations-Meilenstein).

## Restore-Pipeline

`POST /snapshots/:bookId/:id/restore` ersetzt den **gesamten** aktuellen Buchinhalt durch den Stand der Ziel-Fassung. Reihenfolge (Pflicht):

0. **Presence-Lock** — editiert gerade ein **anderer** User am Buch (Live-`page_presence.listForBook`, stale-gefiltert >90 s; eigene Presence auch auf anderen Geräten zählt nicht), gehen dessen parallele Writes während Wipe/Recreate verloren → `409 BOOK_BUSY` mit `editors:[names]`, **bevor** irgendetwas angefasst wird. Der Client (`restoreSnapshot`) fängt das ab, zeigt eine stärkere `appConfirm`-Bestätigung mit den aktiven Namen und wiederholt mit `?force=1` (übersteuert den Lock). Best-effort — schlägt der Presence-Read fehl, wird nicht geblockt.
1. **Ziel planen** — `content_json` parsen → `validateBookJson` → `planFromNodes(tree)` (gleiche Op-Liste wie der Buch-Import). Defekt/leer → 422 `CORRUPT_SNAPSHOT`, bevor irgendetwas angefasst wird.
2. **Auto-Sicherung** — der aktuelle Stand wird via `_buildSnapshotPayload` als neue Fassung mit Label `__i18n:snapshots.autoBackupLabel__` abgelegt → **Restore ist umkehrbar**. `BOOK_EMPTY` (nichts zu sichern) ist ok; jeder andere Fehler → 500 `BACKUP_FAILED` **ohne** Wipe.
3. **Wipe** — alle Seiten **und** alle Kapitel explizit löschen, dann `bookOrder.clearOrder(bookId)`. **Warum explizit:** `pages.chapter_id` und `chapters.parent_chapter_id` sind `ON DELETE SET NULL` — ein gelöschtes Kapitel entfernt weder seine Seiten noch Sub-Kapitel. Das Order-Overlay muss weg, sonst zeigt es auf gelöschte IDs.
4. **Neu anlegen** — `plan.ops` über die Content-Store-Facade (`createChapter`/`createPage`) in Op-Reihenfolge; `tempId → echte chapter_id`-Map wie beim Buch-Import. Danach baut `ensureTree` die Reihenfolge frisch aus `position`/`priority` der neu angelegten Knoten (`buildFromCurrentState`).
5. **Settings** der Fassung übernehmen (best-effort, `saveBookSettings` + ggf. `setBookEntitiesEnabled`; `allow_lektor_book_chat` bleibt 0 — ACL-relevant, nicht Teil der Inhalts-Fassung).
6. **Publikation** der Fassung zurückschreiben (best-effort, nur wenn `publication_json` gesetzt): `upsertMeta` (Voll-Replace der Titelei/epub_*-Felder) + Cover/Autorfoto per `setCover`/`setAuthorImage` auf den eingefrorenen Stand — fehlt ein BLOB in der Fassung, wird das Live-Bild `clearCover`/`clearAuthorImage` (Restore = auf den eingefrorenen Stand setzen). Umkehrbar, weil die Auto-Sicherung (Schritt 2) den vorherigen Publikations-Stand mitgefroren hat.
7. **Stats-Sync** (`syncBook`).

**Invarianten / Konsequenzen:**
- Restore = „Buch-Import in dasselbe Buch" — teilt bewusst die geprüfte Pipeline aus [lib/book-bundle.js](../lib/book-bundle.js) ([routes/jobs/book-import.js](../routes/jobs/book-import.js)).
- Seiten bekommen **neue** `page_id`s → `page_revisions`, Seiten-Chats, `page_stats` der ersetzten Seiten kaskadieren weg. Inhärent bei einem Voll-Restore; die Auto-Sicherung fängt versehentliche Restores auf, nicht die Historie der ersetzten Seiten.
- Nicht-transaktional (wie der Buch-Import); Einzel-`createXxx`-Fehler werden geloggt, nicht geworfen. Die Auto-Sicherung ist die Absicherung gegen einen abgebrochenen Lauf.
- Interleaving (Top-Page zwischen zwei Top-Kapiteln) geht verloren — `treeToNodes` stellt Top-Pages voran. Dokumentierte Grenze des Bundle-Formats, für Fassungen unkritisch (Struktur + Inhalt bleiben vollständig).

## Frontend

[public/js/cards/snapshots-card.js](../public/js/cards/snapshots-card.js) (`Alpine.data('snapshotsCard')`) + [public/partials/snapshots.html](../public/partials/snapshots.html). `showSnapshotsCard` bleibt im Root (Hash-Router + Exklusivität), fachlicher State in der Sub-Komponente. Registry-Eintrag in [feature-registry.js](../public/js/cards/feature-registry.js) (`key: 'snapshots'`, `minRole: 'editor'`, `requiresPages: true`).

- **Speichern** — Name + Notiz (optional, `data-spellcheck`) → `captureSnapshot`.
- **Liste** — `entity-grid-table` (Standard-Tabellenklasse, sortierbar via `sortableTable`, `persistKey: 'book.snapshots'`). Numerik-Spalten rechtsbündig über `.snapshots-table .snapshots-num` (schlägt die linksbündige `entity-grid-table`-Default-Regel per Specificity). Autor-Chip nur bei geteilten Büchern. Veröffentlichte Fassungen tragen einen „Veröffentlicht"-Badge (`isPublished` / `publishedDateLabel`) in der Fassungs-Zelle.
- **Veröffentlichen** — `togglePublish` (`POST …/publish`) markiert die Fassung als Auflage bzw. entfernt die Markierung; `publishingId` sperrt den Button. Der Publikations-Anker + `publication_json` machen die erschienene Auflage reproduzierbar.
- **Öffnen (Reader, read-only, Diff gegen aktuell)** — `openSnapshot` lädt parallel `GET /snapshots/:bookId/:id` (Fassungs-`content`) **und** `GET /book-editor/:bookId/contents` (aktueller Buchstand). Die Render-Sektionen baut `fromSnapshotTree(content.tree)` aus dem geteilten Stream-Modell [public/js/manuscript-stream.js](../public/js/manuscript-stream.js) (geteilt mit Bucheditor `fromPages` und Share-SSR `fromGroups`); der Reader ergänzt pro Page die Diff-State-Felder (`status`/`renderHtml`). `entry.id` = alte page_id (srcId), gegen die der Inline-Diff matcht. Rendert ein Vollbild-Overlay (Modal-Charakter, Esc/Backdrop schliessen, `closeReader`) im **Bucheditor-Look** — reuse der `.book-editor-*`-Stream-/Block-Klassen aus [book-editor.css](../public/css/editor/book/book-editor.css), rein lesend (kein `contenteditable`, kein Save). Pro Seite ein **Inline-Wort-Diff** gegen die aktuelle Version (Match Fassungs-`srcId` == aktuelle `page_id`): `renderInline(fassungHtml, aktuellHtml, diffLib)` aus [page-revision-diff.js](../public/js/page-revision-diff.js) (einspaltig, `del`=seither gelöscht, `ins`=seither hinzugefügt; gleiche Block-/Wort-Pipeline wie `renderSideBySide`, aber ohne Spalten/Collapse). Seiten ohne aktuelles Pendant → Badge „im aktuellen Buch gelöscht"; seit der Fassung neu hinzugekommene Seiten werden als Fussnote gezählt (`readerAddedSince`). Unveränderte Seiten rendern das rohe Fassungs-HTML. Der Reader-Header zeigt zusätzlich den „Veröffentlicht"-Badge sowie einen **Publikations-Nachweis-Strip** aus `extras_summary` (`readerExtraItems`): Zähl-Tags für den eingefrorenen Weltaufbau-/Lektorat-Stand. CSS: [snapshot-reader.css](../public/css/components/snapshot-reader.css).
- **Export** — im Reader-Header: Schnell-Formate (HTML/EPUB/DOCX/MD/TXT) als Download-`<a>` auf die Sync-Route; **PDF** über eine Profil-Combobox (`/pdf-export/profiles`) → `POST /jobs/pdf-export` mit `snapshot_id` → Polling via `startPoll` → Download `/jobs/pdf-export/:id/file`. Der PDF-Job ([routes/jobs/pdf-export.js](../routes/jobs/pdf-export.js)) baut bei gesetztem `snapshotId` das Bundle via `snapshotToBundle` statt `loadContents`; `scope`/`target` sind dann fix `book`/`interior`. Publikations-Metadaten (Cover/Titelei/epub_*) kommen aus der **eingefrorenen `publication_json`** der Fassung (via `snapshotPublication`), nicht aus dem Live-`book_publication` — ein Export von „Fassung 3" trägt also das Impressum/Cover von damals. Fehlt die eingefrorene Publikation (alte Fassung vor Migration 235, oder Buch ohne `book_publication`-Zeile), fällt der Job auf die Live-Daten zurück. Gilt gleichermaßen für den EPUB-Job und den synchronen Sync-Export ([lib/export-send.js](../lib/export-send.js)#`buildExportMeta` nimmt `{ publication }`). Dedup-Key trägt `:snap<id>`.
- **Wiederherstellen** — `restoreSnapshot` mit `appConfirm(danger)`-Modal (kein natives `confirm()`); bei `409 BOOK_BUSY` (anderer User editiert gerade) eine zweite, stärkere `appConfirm` mit den aktiven Namen (`snapshots.busyConfirm`), dann erneut mit `?force=1`. Nach Erfolg `app.loadPages()` + Liste neu laden (Auto-Sicherung erscheint dort). `restoringId` sperrt die Zeile.
- **Löschen** — `deleteSnapshot` mit `appConfirm(danger)`-Modal; bei einer veröffentlichten Fassung antwortet der Server `409 SNAPSHOT_PUBLISHED` → stärkere `appConfirm` (`snapshots.deletePublishedConfirm`), dann erneut mit `?force=1`.
- **Vergleich** — zwei Comboboxen (Default: zweitneueste vs. neueste), Buch-Level-Diff via [book-snapshot-diff.js](../public/js/book-snapshot-diff.js) (`diffSnapshots`): Summen-Badges (hinzugefügt/entfernt/geändert/umbenannt/verschoben) + Zeichen-Delta, pro geänderter Seite lazy ein Wort-genauer Side-by-Side-Diff via [page-revision-diff.js](../public/js/page-revision-diff.js)#`renderSideBySide`. Zusätzlich ein **Publikations-Metadaten-Diff** (`diffPublication` derselben Datei) über die eingefrorenen `publication`-Text-Felder beider Fassungen (Autor/ISBN/Untertitel/Jahr/Verlag/Reihe/Widmung/Impressum/Copyright/Titelei-Text/Bio + Cover/Autorfoto vorhanden-ja/nein) — genau die publikationskritischen Felder, die der Inhalts-Diff nicht sieht. Als eigene Tabelle (`snapshots-pubdiff`), nur bei Änderungen sichtbar.
- `__i18n:`-Label-Marker (z.B. Auto-Sicherung) werden in `fassungLabel` via `_resolveLabel` in der Locale des Betrachters aufgelöst.

CSS: [public/css/components/snapshots.css](../public/css/components/snapshots.css) (nur snapshot-spezifische Tweaks; Tabelle nutzt `entity-grid-table`, Diff-Zellen `revision-diff-*` aus `page/page-revision-viewer.css`). i18n-Keys: `snapshots.*` in [de.json](../public/js/i18n/de.json) / [en.json](../public/js/i18n/en.json).
